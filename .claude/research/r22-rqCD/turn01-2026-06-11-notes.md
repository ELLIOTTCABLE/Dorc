## Findings
- BUCK2 IS d-1's LIVING PROOF-OF-CONCEPT: `buck2 log` is an always-on, per-invocation durable event log with ~15 query lenses over ONE producer — `what-ran`, `what-failed`, `what-materialized`, `what-uploaded`, `critical-path`, `slowest-path`, `replay`, `show`/`show-user`, `summary`, `diff action-divergence`, `cmd`, `what-up`. Exactly d-1's "one producer, many lenses." [B-buck2-log-2024]
- Buck2 log has STRUCTURED MACHINE-READABLE output baked in: `--format readable|tabulated|json|csv`, plus `show` (JSON-per-line) and `show-user` (JSONL user-event-log). The dump is queryable, diffable, and replayable — not just a flat text trace. `diff action-divergence` is precisely "find the first divergent action between two builds … useful for identifying non-determinism" — a built-in golden-trace-style diff lens. [B-buck2-log-2024]
- COUNTER-THESIS, FIRST-PARTY: Bazel core dev (lberki) on the always-on-debug-log idea: `aquery` "requires extra conscious effort"; "execution logs are too large to be always uploaded"; "BEP does not contain enough information." This is the capture-without-consumer / friction risk stated by the people who ship it. The fix they shipped (Bazel 7.1 `--execution_log_compact_file`) is about MAKING the durable cheap-enough-to-keep, implying the naive durable was NOT. [B-bazel-issue18643-2023]
- rq-D HAND-ROLLING PITFALL CATALOGUE (bazel_conduit, Rust BEP->OTel converter): 11 lessons-learned, several are exactly Dorc's determinism concern. UUID->TraceID derivation (1 trace = 1 invocation). Stale clock: `BuildStarted.start_time` can be the *server* start time (weeks old), not the invocation — they fall back to event-emit time. Epoch-1970 bug: `start_time_nanos=0` + duration-as-end produces 1970-start multi-day spans; they reject timestamps below a ~2001 threshold. Clock skew -> negative/zero durations (action-level unreliable; target-level derived is better). BatchSpanProcessor silently DROPS spans past queue=2048 (they raised to 65536). [B-bazelconduit-readme-2026]
- rq-D SECRET-LEAK is a real, designed-for hazard in span emission: conduit ships an in-process scrubber that runs BEFORE any attribute is set on a span, because command-lines / `--client_env` / workspace-status routinely carry CI tokens that otherwise land verbatim in span attrs -> backend. Defense-in-depth via Datadog `replace_tags` / OTel Collector `redactionprocessor`. Directly relevant to Dorc's provenance-plane-as-leak-surface. [B-bazelconduit-readme-2026]
- rq-D "minimal OTel, ideas not machinery" datapoint: conduit KEPT the span model + semantic-convention-style attribute namespace (`bazel.<component>.<field>`, ~100 typed constants) + W3C-ish trace identity, but it's a *sidecar* that ingests an existing event stream rather than instrumenting Bazel with the SDK in-process. The build tool itself emits a neutral event stream (BEP); OTel mapping is downstream. That's an architecture lean worth flagging for Dorc: emit a neutral durable, convert to OTel at the edge.

> Bazel provides ample tools to query what it would run (`aquery`) and what it did (`--execution_log_*_file` or the BEP) but neither of these is amenable to post-facto debugging:
> * `aquery` requires extra conscious effort
> * The execution logs are too large to be always uploaded
> * The BEP does not contain enough information
> So it would be nice to be able to do some sort of logging to answer questions like "why did this build done two days ago did not reuse the cached actions from that other build done 30 minutes before it".

> Lesson 5: BatchSpanProcessor Can Silently Drop Spans … default queue size is 2048 … overflow this easily … queue 65,536
