## Findings (lifted, most load-bearing first)

- rq-H WINNER MECHANISM: rustc's completeness gate is NOT a type-system trick — it's a
  **tidy check that regex-greps `compiler/` for `\bE\d{4}\b` and cross-checks against the
  registry both directions** (every registered code must be emitted; every emitted code
  must be registered). This is *exactly* the Pottier-direction gate Dorc wants, and it is
  cheap, plain, and adoptable in a plain Rust workspace as a test/CI step. [A-rustc-tidy-errorcodes-2026]
- rq-H: the gate has FOUR stages, all enforced in CI: (1) extract registry list, (2) each
  code has a long-form `.md` explanation w/ a `compile_fail` doctest using its OWN code,
  (3) each code has a UI test (`Exxxx.rs` + `Exxxx.stderr`) that contains its own code,
  (4) each code is actually emitted by the compiler (regex). [A-rustc-tidy-errorcodes-2026]
- rq-H ROT EVIDENCE (direct): the gate carries TWO hardcoded allow-lists —
  `IGNORE_DOCTEST_CHECK` (5 codes) and `IGNORE_UI_TEST_CHECK` (7 codes) — each commented
  "This list will eventually be removed." This is the visible rot/escape-hatch: codes that
  predate the rule are grandfathered, and the backlog persists for years. The *direction of
  travel* is monotone-shrinking (you can't add to it without a reviewer noticing), which is
  the key design property. [A-rustc-tidy-errorcodes-2026]
- rq-B retire-don't-delete: a *separate* tidy sub-check diffs against the base commit and
  HARD-ERRORS if any `error_codes/*.md` file is deleted ("Error code explanations should
  never be removed!"). Retired codes get a `#### Note: this error code is no longer emitted`
  marker instead, which exempts them from stage-3/4. Codes are append-only & monotonic;
  numbers are never reused. [A-rustc-tidy-errorcodes-2026] [A-rustc-devguide-errorcodes-2026]
- rq-B registration workflow (who/how): adding a code = find next free number (`rg E0806`),
  write `E0806.md` explanation, add `0806,` to the `error_codes!` macro in numeric order,
  emit via `struct_span_code_err!(dcx, span, E0806, msg).emit()`. No central registrar —
  any contributor does it, the tidy gate + reviewer enforce correctness. [A-rustc-devguide-errorcodes-2026]
- rq-H ErrorGuaranteed: a **zero-sized type unconstructable outside `rustc_errors`**,
  produced only by `.emit()`. Possessing a value statically proves "an error was already
  reported → compilation will fail." TWO caveats that bound its use: (a) carries NO info
  about error *kind* (could be a delayed-bug), so don't branch emit-decisions on it; (b)
  means "already emitted," never "will emit." [A-rustc-devguide-errorguaranteed-2024]
- rq-B Fluent/translation STALLED (counter-thesis confirmed): as of Oct 2024 the dev-guide
  states the infra "causes friction," is "pending a redesign," has "no current active
  redesign proposals," and is explicitly **NOT mandated** — contributors may sidestep it.
  The piece that *did* stick: `#[derive(Diagnostic)]` does compile-time Fluent validation
  (parse errors + duplicate-message-id detection at compiler-build time). [A-rustc-devguide-translation-2024]

> ErrorGuaranteed is a zero-sized type that is unconstructable outside of the rustc_errors crate. It is generated whenever an error is reported to the user, so that if your compiler code ever encounters a value of type ErrorGuaranteed, the compilation is statically guaranteed to fail.
> * It does not convey information about the kind of error ... you should not rely on ErrorGuaranteed when deciding whether to emit an error
> * ErrorGuaranteed should not be used to indicate that a compilation will emit an error in the future. It should be used to indicate that an error has already been emitted
