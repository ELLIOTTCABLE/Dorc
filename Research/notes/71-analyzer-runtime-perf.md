# Phase 1 — analyzer runtime performance: the cliffs, the bounds, what keeps Dorc off them

Mining the on-disk corpus for *cost* (not architecture). All bounds below are from primary sources (paper on disk, or peer-reviewed result located + graded A). The throughline: **the cost cliffs in program analysis are well-mapped, and Dorc's design choices either sit safely below them or walk up to one deliberately. The single most dangerous knob is context-sensitivity.**

## The three complexity cliffs (know exactly where they are)

1. **Cubic is the floor for this *kind* of analysis — and that's OK.** Heintze–McAllester 1997 ("On the Cubic Bottleneck in Subtyping and Flow Analysis", LICS; Cornell `kozen/Papers/cubic.pdf`, grade A): a large class of analyses — CFL-reachability, set-constraints, flow analysis, Andersen points-to — are **2NPDA-complete**, and 2NPDA's best known algorithm is **cubic**. Sub-cubic has resisted decades of attack. Andersen points-to is canonically **O(n³)** in #variables. → Dorc's may-mutate/reachability layer is CFL-reachability (notes 52/53), so **expect cubic worst-case per analyzed unit and stop trying to beat it.** This is *fine* because the unit is a role/function (small n), not the whole corpus (§scale below).

2. **IFDS is O(ED³) — but collapses on the realistic case.** Reps–Horwitz–Sagiv 1995 (on disk, line 3): tabulation "runs in time **O(ED³)**" where E = supergraph edges, D = dataflow-fact domain. The D³ is the scary term — *but* for **h-sparse** problems (each statement touches a small slice of facts — the normal case, and true for shell commands: one command touches `pkg:foo`, not all facts) it **collapses toward O(ED)** (note 52). → Dorc's fact domain per unit is small and sparse; the cubic-in-D term is a worst-case that real provisioning scripts don't hit. *Design rider:* keep transfer functions sparse (a command's effect names specific facts), and D stays small.

3. **Context-sensitivity is the EXPTIME cliff — but the paradox says *Dorc's data model* decides whether it bites.** Van Horn–Mairson 2008 ("Deciding k-CFA is complete for EXPTIME", ICFP; grade A): **k-CFA for k≥1 is EXPTIME-complete.** 0-CFA is polynomial (cubic); **1-CFA already crosses into provable exponential worst-case.** → directly arms the AGENTS *"context-sensitivity dial"* decision: **it is not a precision tuning knob, it is the polynomial↔exponential boundary.**
   - **The k-CFA paradox (Might–Smaragdakis–Van Horn, PLDI 2010; grade A) — the load-bearing refinement.** The *same* k-CFA spec is **EXPTIME for functional languages but polynomial for OO ones.** Why: the exponential blowup comes from **environments that combine variables drawn from distinct calling contexts** (closures capture and recombine environments); OO objects don't recombine contexts that way, so they stay polynomial. → **The redline is not "context-sensitivity is fatal" — it's "context-sensitivity is fatal *iff* your abstraction lets contexts combine closure-style."** Dorc's abstract state is a **flat map of system-state facts** (`pkg:foo present@X`, `file:/etc/y exists`), not nested closures capturing other contexts' environments. GUESS (important, verify in the spike): **Dorc's flat-fact model is in the polynomial-amenable / OO-like regime**, so **per-host and per-role-invocation context-sensitivity may be affordable**, not fatal — which matters because the analysis-architecture flagged per-host context as a *precision dial* (it's likely a usable one). The discipline that keeps it safe: **never introduce a context construct that lets one unit's context capture-and-recombine another's fact-environment** (that's the closure pattern that detonates). Default still 0-CFA/per-unit-summary; add per-host/per-role context where it buys probes, having confirmed the fact-domain stays flat.
   - The TAJS evidence (below) confirms context-sensitivity dominates cost in practice, not just theory — but TAJS analyzes JavaScript (closures, the exponential case); Dorc's flatter domain is a friendlier starting point.

## The memory wall (it's RAM, not time, that kills real analyzers)
- **TAJS** (Jensen–Møller, on disk lines 820–829, grade A): the prototype **ran out of memory at a 512 MB limit** on the larger benchmarks; the authors state "analysis speed and memory consumption have not been key objectives" yet still hit the wall; and **"Using recency abstraction is crucial: it generally reduces memory consumption"** (and craters precision when disabled, 87%→<2%, note 54). → For Dorc: **recency strong/weak abstraction is a *memory* lever as much as a precision one** (already adopted in analysis-architecture; this is a second, independent reason to keep it). Memory, not wall-clock, is the likely first wall at corpus scale.
- **Datalog/Doop** (Bravenboer–Smaragdakis, on disk: "memory consumption" hits at lines 1346, 2192; "single non-reducible relation can become a bottleneck" line 2084): the declarative route **materializes all facts**, so memory is the scaling constraint — a fully-materialized fact base for a huge corpus can blow RAM. **Soufflé quantifies the wall** (on disk lines 1009–1032): "millions of variables… billions of tuples" *but on a* **128 GB-RAM Xeon server**, with a **"huge performance gap"** vs a 32 GB desktop — i.e. the materialized approach is fast *only when it fits in RAM*, and the working set is large. Soufflé spends explicit engineering (B-tree vs trie, "reduced memory consumption by ≈7 GB / up to 59%") fighting exactly this. → This is the **hidden cost of the "queryable fact base" extensibility** (analysis-architecture decision #1, Datalog-vs-IFDS-vs-hybrid): Datalog buys extensibility/provenance but **pays in materialized-memory** (and is RAM-fit-sensitive); IFDS/demand computes only what's queried. The hybrid's burden is keeping the materialized layer bounded. *Flag for the substrate spike: measure peak RSS, not just runtime, and check the working set fits commodity RAM (the ops corpus is far smaller than OpenJDK, so this likely holds — but verify, don't assume).*
- **SDG slicing can be exponential** (Horwitz–Reps–Binkley, on disk lines 1928, 2645, 3123: "may require time exponential", "the cost can increase by an exponential factor"): more precise slices cost exponentially more. → Slicing for sub-host precision (1B / Tier-B) must be **bounded/demand-driven**, never "compute the most precise slice."

## The scale answer is structural, and already correct (don't re-derive — confirm the numbers)
- **Facebook/Infer** (Distefano et al., on disk, grade A): **100M LOC analyzed in < 30 min on a 24-core server**; ~**1 trillion LOC/day** across the fleet; compositional ⇒ "**naturally incremental — changing one** [procedure re-analyzes only it]" (line 1034); **70% fix-rate at diff-time vs near-0 batch** (lines 156–160). → The scale lever is **compositional per-unit summaries + diff-time incrementality**, and the numbers say it works at corpus sizes orders beyond any plausible ops corpus. The ops corpus is *embarrassingly modular* (R roles × H hosts) — a far easier shape than 100M-LOC Hack. **Dorc's scale risk is not the corpus size; it's accidentally choosing a non-compositional or context-sensitive design that forfeits this.**
- **Per-host is instantiation, not re-analysis** (IDE summaries; note 53): H hosts ≈ H cheap summary-instantiations, not H full analyses. The expensive thing (analysis) is per-*role* and cached; the per-host cost is the cheap thing.
- **Incremental recompute is ~80%-cheap, measured** (Reviser / Arzt–Bodden, on disk lines 36/145/155, grade A): incremental IFDS/IDE via clear-and-propagate **"saves up to 80% vs full recomputation"** (65–80% across real Soot test cases), *with identical results* — over-approximating the affected set keeps it safe. → Confirms the diff-time story with a number: re-analyzing a changed role is ~1/5 the cost of cold analysis. The compositional route (Facebook) gets this *for free* (change one role → re-summarize only it); Reviser is the bolt-on for the IFDS route. Either way, **diff-time recompute is cheap enough to be the primary mode.**

## The footgun list (phase-1-specific "wish I'd known")
1. **Reaching for context-sensitivity to fix precision** → EXPTIME. Mitigation: 0-CFA default; selective/demand context only; treat any global context-sensitivity proposal as a redline. (This recasts the AGENTS open-decision as a *safety* boundary, not just a tuning choice.)
2. **Choosing the Datalog-everything substrate without a memory budget** → materialized-fact RAM wall. Mitigation: hybrid (IFDS/demand core + bounded relational layer); measure RSS in the spike.
3. **Dropping recency abstraction** → both precision collapse *and* memory blow-up (TAJS). Mitigation: it's non-optional; it's load-bearing on two axes.
4. **Most-precise slicing** → exponential. Mitigation: demand/bounded slicing only.
5. **Serializing the IR/summary format late** → can't incrementalize or distribute cached summaries across an org. Mitigation: the on-disk summary format is a public seam — **version it from commit 1** (the analyzer's own "diff-time" needs cached prior summaries; so does sharing oracle/role summaries across a team). **But cache *persistence* is genuinely hard — temper the optimism**: rust-analyzer (the incremental-computation poster child, on Salsa) **deliberately does NOT persist its cache to disk** — "opening a project means waiting a dozen seconds while [re]processing the standard library"; persistent caching is a *known-hard, still-open* problem (rust-analyzer issue #4712), because serializing a query-graph with correct cross-run invalidation is subtle. → For Dorc: **version the format now (cheap, right), but treat persistence as an *optimization with a cold-start fallback*, not a v1 given.** Dorc's saving grace vs rust-analyzer: its primary unit is the *diff* and its corpus is far smaller, so cold-start recompute may be acceptably fast for v1, with persistence added when the org-scale corpus makes cold-start hurt. (Salsa's *durability* idea — inputs LOW, interned-values HIGH, so high-durability facts survive more edits — is a useful design cue for which summaries to cache hardest.)
6. **Whole-program (batch) mindset** → forfeits diff-time incrementality, which Facebook shows is *the* deployment lesson (0%→70%). Mitigation: compositional + incremental from the start; the analyzer's primary mode is "analyze the diff," not "analyze the world."

## Net for the perf architecture (phase 1)
The analyzer cost is **bounded and well-understood**: cubic-per-unit (acceptable; units are small and sparse), made fleet-scalable by **compositional summaries + diff-time incrementality** (Facebook-proven at 1000× the scale), kept off the **EXPTIME cliff** by defaulting to **context-insensitivity** (selective context only on demand), and kept under the **memory wall** by **recency abstraction + a bounded (not fully-materialized) fact representation**. The connection to the perf *thesis* (note 70): every one of these is a place the analyzer can be deliberately *less precise to stay cheap*, and every such choice is **elision-safe** (cost = more probes/runs, never a wrong skip) — so the analyzer can be tuned aggressively toward speed, in the safe direction, exactly as the perf reframe predicted.
