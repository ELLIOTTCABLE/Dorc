# Ansible user-study synthesis (Carreira et al. 2025, mixed-methods: 59,157 posts + 20 interviews)

Source: arXiv 2504.08678v2, "The Ultimate Configuration Management Tool? Lessons from a Mixed Methods Study of Ansible's Challenges" (Carreira, Saavedra, Mendes, Ferreira; CMU/IST-Lisbon/Porto, Oct 2025). **Grade A** [A-ansible-challenges-mixed-methods-study-2025] — peer-reviewed, empirical, recent, replication package on Zenodo (10.5281/zenodo.15031088). The single strongest user-study source found; it reads almost like a validation spec for Dorc.

## RQ1 — the empirical challenge taxonomy (7 themes, descending prevalence of recent posts)
1. **Control/Data Flow & Execution — 37.9% (the #1 pain).** Loops 16.4% (*"I don't understand why loops are so hard to do in this language"*), SSH-connection 14.9%, Variable Management 14.4%, Conditional Execution 12.4%, Undefined Variables 12.3%. Interviews: sequential execution slows runs + **"lack of synchronization primitives"**; dependency-management complexity; **stateless = double-edged ("simpler to run but lacking built-in features for preserving or comparing states")**; push/agentless praised (low overhead) *and* criticised (complicates continuous enforcement).
   → **Strongest validation of Dorc's thesis yet.** The #1 real-world Ansible pain is exactly the category a *real shell-superset with real control flow* deletes (loops/conditionals/variables become native), and "no built-in state preserving/comparing" + "no synchronization primitives" is precisely what Dorc's analysis-derived state/effect model supplies.
2. **Reliability — 32.3%.** Error Handling 16.4%, file management 11.5%, permissions 8.1%, JMESPath 5.9%, shell-commands 5.4%. Interviews: **"Ansible is hard to debug" (n=10/20)**; I7: *"if there's more complex stages... I transfer those to a shell script that I can pass outside of Ansible and test separately"*; B2: *"stuck for 15 minutes in a supposedly 1-min script... no visible changes on the terminal"* (the buffering/opacity pain); cryptic module errors; B4: *"there's not a proper way to debugging this [IaC]"*.
   → Validation: people **escape to shell to debug** (Dorc *is* shell); realtime streaming output + good errors (planning-log hard requirements) target the #2 theme directly.
3. **Lack of Support — 32.1%.** AWX 20.9%, module-execution-failures 12.7%, integration 10.8%, module-usage 9.8%, Windows 9.1%. E4: *"lack of support for some modules — you often have to write your own or hope someone did it."*
   → The module-coverage/bootstrap problem (planning-log existential risk) confirmed from the user side. Dorc's "paste raw upstream shell, gradually enhance" beats "hope someone wrote a module."
4. **Usability — 16.6%.** **Jinja2 33.1% (*"incredibly challenging to work with"*)**, quoting 22.4%, playbook-inputs 20.4%, YAML 10.8%, regex 8.2%. E5: *"I once spent 8 hours figuring out the escaping on a line of SQL because YAML is so bad at quotes... The hell that is escaping in YAML will never die."* E8: declarative *"looks weird and alienating"* coming from imperative scripting (mental shift, n=4).
   → MASSIVE validation: YAML+Jinja escaping/quoting is a top usability pain; Dorc (real sh-superset, no YAML/Jinja layer) deletes the entire category — the "better at what Ansible does poorly" thesis, empirically backed.
5. **Performance & Scalability — 7.5%.** dynamic-inventory 31%, inventory-mgmt 29.8%, performance 17.7%, remote-exec 8.5%, delegation 7.7%. Interviews: slow execution (n=8); E4: *"It does not scale when you manage 100s of servers, it becomes slow because there is no agent"*; B1: *"run a playbook for ~100 hosts, it gets really confusing trying to make sense of the messages."*
6. **Maintenance — 3%.** deprecated-features 34.4%, dependency-mgmt 23.3%, compatibility 19.6%, versions 14.2%, version-control 12.3%. Interviews — **THE deep structural critique for Dorc**: *"removing configurations is difficult (n=3)"*; E2: *"removing config from hosts tends not to work so well in Ansible"*; I3: **"Deletes are hard: one cannot merely delete parts of the Ansible playbook and expect those changes to be undone in hosts. Instead, we must change or even add tasks undoing the configuration we want gone."** Config drift (n=5).
   → Declarative-convergence-without-state **cannot handle deletion/drift** — there's no record of what was established, so nothing knows what to undo. Dorc's establish/kill-fact dependence model is exactly what makes "delete the task → the effect is reverted/flagged" tractable. Concrete differentiating capability target.
7. **Information Gaps — 2%.** Documentation (99.8% of the theme). Misleading community advice; steep curve for advanced features.

## RQ2 — what drives adoption / what people LOVE (the sacred cows)
- **Ease of use / low barrier** (easy n=13, easy-onboard n=7, low-setup n=6): *"I only really needed to understand a small set of concepts."* **Readable syntax** (n=6) — note the **YAML tension**: loved for readability, hated for escaping.
- **Agentless / SSH-only (n=9) — the headline comparative advantage** over Puppet/Chef/Salt. E5's anecdote: Puppet paged the admin at 3am 3× (restored broken config, then self-updated from git) — agentless+push avoids the agent-daemon hell. (Dorc keeps this: ship portable shell over ssh, zero target deps.)
- **Idempotence** (n=2): I3 *"some commands are dangerous to invoke twice."*
- Flexibility / **one-tool-does-it-all** (n=2; heterogeneous Win/Linux/network), Compatibility (n=6), lightweight/cost-effective (*"inexpensive [Kubernetes] for scalability"*), extensible/Python (n=4).
- **Community / popularity** (n=4+4); often **mandated** by org/school (n=8) or recommended (n=3).
→ Dorc must preserve: agentless-ssh, idempotence, low-barrier "small set of concepts," readability (without YAML's escaping tax), heterogeneity, extensibility. The "mandated/popular" driver is the adoption moat Dorc must overcome (community = the planning-log existential risk again).

## RQ3 — what they wish for (a near-literal Dorc feature list)
- Documentation (n=10); **better debugging tools (n=5) + better error messages (n=6) + rollback-on-failure (n=5)**.
- **Performance: speed (n=7, *"slow runs is an eternal pain point"*) + parallelization (n=4)**; Mitogen for SSH; B1's **tag-based parallel groups** (*"tasks with tag A can run in parallel with tag B"*) — i.e. they're hand-asking for the dependency-derived parallelism Dorc *computes*.
- **Advanced programming capabilities**: high-level concepts (n=3: *real loops, complex conditionals, synchronization primitives*) + **state management (n=5)**. E7: *"even though Ansible claims to be completely stateless, which is supposed to be its strength, I'm not so convinced. Without state management, it's challenging to handle certain problems."*
- Tiered UX: beginners want onboarding + rollback; experienced want speed + parallelization.

## Discussion → reads like Dorc design guidance
- **Debugging**: IaC lacks the debugging utilities of C/Python; practitioners *"resorted to shell scripts instead"*; tools should give comprehensive logging, step-by-step debug, **automatic rollback (save prior state or compensating ops)**, and *mandatory logging standards for community modules* (≈ Dorc's oracle-authoring contract).
- **Syntax**: YAML *"highly complex and error-prone"* (Ansible AND k8s). Cited directions: replace YAML w/ TOML; **Pulumi (IaC in real imperative languages while keeping declarative features)**; Nix (functional). *"to introduce higher-level language concepts, we must go beyond configuration languages"* — this is the Dorc bet, stated by the researchers.
- **Performance**: **Jetporch (Rust Ansible-alternative, avoids Python overhead)** is the cited direction; refactor for parallelization + SSH optimization + partial execution.
- Tiered UX + universal documentation-gap problem.

## Net for the three spaces
- **Space 3 (user-study)**: Ansible side is now richly covered. Failures: control-flow expressivity, debugging opacity, YAML/Jinja escaping, scaling/perf, **deletion/drift (no state)**, module gaps. Sacred cows: agentless-ssh, idempotence, low-barrier/readable, one-tool heterogeneity. Costly long-run choices: YAML+Jinja as the language; Python execution model (perf); statelessness (deletion/drift/debugging). *(Still need the Terraform side + raw HN/Lobsters sentiment for balance.)*
- **Space 1 (corpus)**: the academic IaC datasets (Opdebeeck/Rahman/GLITCH/Begoug + this study's Zenodo package) are curated, labeled corpora — a faster bootstrap than scraping GitHub raw, and they come with defect/smell labels usable as validation criteria.
- **Space 2 (go/no-go)**: the loved features (agentless-ssh, idempotence) and the wished features (rollback/state, parallelization-from-dependencies, debugging/logging) sharpen which functions Dorc should integrate vs cede.
