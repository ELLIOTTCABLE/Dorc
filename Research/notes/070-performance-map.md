Round 4. The prior three rounds (feasibility, analysis architecture, userbase) never treated **performance** as its own axis. This round does, across the three phases the user named, each with a different literature and a different failure mode.

## The reframe that orients the whole round (the load-bearing insight)
The user's intuition — "all program-analysis is a tradeoff between performance and usefulness while staying *just short* of giving up soundness, so perf accounting matters to correctness/safety" — is **half right, and the half that's wrong is the important half for Dorc.**

- For an **under-approximating bug-finder** (Infer's normal mode), cutting corners for speed creates **false negatives** = missed bugs = unsound *for its purpose*. There the triangle is real: perf ⟂ soundness.
- Dorc is a **sound over-approximator**. Cutting corners for speed costs **precision**, which in Dorc means **more probing / more re-running** — *slower deployment, never a wrong one* — **provided every shortcut fails toward the conservative end of whichever soundness governs that phase.**
- So for Dorc the triangle **collapses to perf-vs-precision** — but the safe direction is **phase-dependent, not a single ⊤** (correcting an earlier draft of this note that said "always fail to ⊤": the two soundnesses have *opposite* fail-safe actions, so one ⊤ cannot serve both):

  > **Probe-phase shortcuts fail toward *don't-execute*** — an un-classifiable leaf is *un-probeable* (defer it to apply); never run it raw in the read-only pass. **Apply-phase shortcuts fail toward *do-execute*** — an un-provable skip ⇒ run it. The only way a perf decision becomes a soundness (outage) bug is failing the *wrong* way for the phase: running a maybe-mutation in the probe, or skipping a maybe-needed mutation at apply. Enforce per-phase; property-test both kernels (`inert ⇒ no-mutation` for probe; `skip ⇒ proven-converged` for elision).

The first "wish I'd known": **you're allowed to be sloppy for speed — but in *opposite* directions in the two phases.** It's still *liberating* for performance — Dorc can aggressively time-box, sample, and approximate where a bug-finder cannot, because every degraded mode is the safe-but-slow direction (probe-more / run-more), never "miss a mutation."

Two consequences threaded through the round:
- **The probe is real shell on real hosts.** Probing is "free" for *analysis* soundness but **not free for the fleet** — an expensive oracle check fanned across 10⁴ hosts is itself a load event. Precision still matters *economically* even though it's correctness-safe; oracle checks need a cost budget. Net: perf is a **cost-and-latency** problem for Dorc, not a correctness one — chase it hard (for adoption); the only thing to *fear* getting wrong is the per-phase fail-direction.
- **Soundness is relative to the oracle contract** — read-only-ness is *demanded + assumed + best-effort-checked*, not guaranteed. A lying oracle is GIGO, outside the bound; don't over-architect against it or exaggerate its failure (the inert↔effectful boundary is still where defensive testing concentrates, because *our own* mis-derivation there breaks both soundnesses).

## How to research this well (the up-front plan the user asked for)
1. **Phase 1 — mine, don't re-fetch.** I already deep-read these papers for *architecture*; re-read them for *cost*: complexity bounds, measured timings, memory numbers, where each says "this is the wall." Targeted external search only for the named cost-model results (cubic points-to bottleneck / Heintze–McAllester, k-CFA-is-exponential / Van Horn–Mairson, "memory not time is the bottleneck" practitioner lore, Datalog space cost). Grade as before.
2. **Phase 2 — go to the source code and the ops war-stories.** Ansible's slowness and the mitogen/pipelining/strategy responses are *documented prior art* (someone hit the wall and wrote it down). pyinfra markets fast-parallel as its thesis — read *why* it's faster. Read sshd's own `MaxStartups`/`MaxSessions` defaults (the hard ceiling nobody mentions until it bites). Capture concrete numbers (default forks=5, fd limits, control-socket behavior).
3. **Phase 3 — survey the mature tools precisely, since this is the "not my domain, easy to ignore" phase.** Catalog exactly what Ansible/Terraform/k8s give for ordering, batching, partial-failure, and *why* (each knob is a scar). Map each to Dorc's derived-dependence-graph model: Dorc *computes* the DAG the others make you author — but executing a DAG across a constrained fleet is a **scheduling** problem (list scheduling, critical path, resource-constrained), which is its own literature.
4. **Cross-cutting — hunt the lock-in traps** ("things I'll wish I'd known"): architectural choices other tools made early that capped their performance ceiling and couldn't be undone after users depended on them. Seed list below; grow it.
5. **Grade every source** (peer-reviewed/primary/source-code A; practitioner war-story B; SEO/AI listicle rejected). Reproduce durable findings in notes 71/72/73. Synthesize into `plans/076-performance-architecture.md` only after coverage feels honest per phase.

## Seed list of lock-in traps to confirm/refute (the "wish I'd known" hunt)
- **Terraform: monolithic state at the center → 15–30 min plans, contention queue, secrets-in-state** (already evidenced, notes 62). Lock-in: state *is* the architecture; can't decentralize after the fact. Dorc dodges by probing real state — *verify there's no equivalent hidden central bottleneck we'd accrete* (e.g. a summary cache that becomes a contention point).
- **Ansible: fork-per-task + per-task SSH + linear strategy** → slow by construction; mitogen/pipelining/`free`-strategy are *retrofits* that never fully fixed it because the execution model was fixed early. Lock-in: the controller-push fork model. (Dorc's compositional-summary parallelism must be designed in, not bolted on.)
- **Whole-program vs compositional**: tools built whole-program (batch) couldn't later become diff-time/incremental without a rewrite (Facebook's explicit lesson). Lock-in: compositionality is an *architectural* property, retrofit-hostile. → Dorc must be compositional + incremental from commit 1.
- **Context-sensitivity / IR granularity** caps the precision-vs-perf ceiling and is expensive to change once summaries are serialized and cached on users' machines. The on-disk summary/IR format is a public seam → version it from day 1.
- **Datalog-everything** materializes all facts → memory is the wall at corpus scale (vs IFDS demand). Choosing the substrate (notes 53 decision #1) is partly a *memory-ceiling* choice, not just an extensibility one.
- **Probe/oracle cost ignored**: if oracle checks aren't cost-budgeted, embarrassingly-parallel probing becomes embarrassingly-parallel *load* (DDoS-your-own-fleet / your-own-apt-mirror). Design a per-probe cost class early.

## The probe optimizer is *behaviour-dropping* (the framing correction)
A normal optimizing compiler / query planner is **semantics-preserving** — it may reorder or batch, but every behaviour must still happen. Dorc's probe generator is a **behaviour-*dropping*** optimizer: it may legally *omit entire computations* (the un-provably-inert ones, deferred to apply), because the probe's contract is not "same observable behaviour" but "a provably-read-only *subset* + a verdict." It over-optimizes in the one direction a normal optimizer cannot. Mental model: *an optimizing compiler that is allowed to delete code it can't prove safe to run.* The two legal moves per check/subtree: **hoist-and-batch** (cheap, independent, read-only → lift into the flat parallel probe; the 40-`dpkg`→1-`dpkg -l` win) and **keep-under-guard** (expensive → retain a cheap local guard so it's elided when irrelevant; don't `curl` a remote service when `[ -f /etc/flag ]` is false). Both compose with ~zero controller cost because retained guards run *locally on each host's probe*. The choice between them is **cost-driven** — hence the cost model below.

## The cost model (newly surfaced; the user's `.perf-info` question)
Over-probing is **correctness-safe but economically central** — cutting unnecessary expensive work is the entire product, so "just flatten everything and probe evenly" is *not* an acceptable punt. The optimizer needs a per-check cost signal, and the UX constraint is hard: we are **not** a YAML metadata blob and must not burden the engineer with mandatory annotation (that breaks the deployer gradient). Open design space (research target — DB optimizers + PGO are the prior art):
- **Static cost-class hint** (optional): an oracle *may* declare a coarse class (`local-stat` / `local-exec` / `network` / `remote-rpc`) — cheap to author, optional (absent ⇒ assume expensive = keep-under-guard, the safe-for-perf default that never *flattens* a possibly-expensive thing into the even bag). A separate `.perf-info` callable is probably over-engineering vs a one-line declaration; evaluate.
- **Profile-guided** (the likely winner): learn costs from probe telemetry across runs (every probe already measures itself — the realtime-output requirement gives this for free). Adaptive, no annotation, mirrors DB runtime re-planning / AutoFDO. Cold-start uses the static hint or the conservative default.
- **Structural**: a guard that gates a large subtree is worth keeping *regardless* of its own cost (it elides much) — a critical-path / expected-work calculation, not a flat threshold.
- **Soundness rider**: keeping a guard to gate an expensive check is only valid if the *guard itself* is read-only **and** evaluable against initial host state (no dependence on an upstream unapplied mutation) — same purity + intra-run-independence conditions as everything else; don't let the cost model retain an order-dependent guard that then mis-gates.

**Tuning-parameter sensitivity (the user's flag).** Perf is sensitive to a handful of knobs that live at the intersections of the three dualities: batch-vs-guard threshold; probe fan-out width (phase 2); apply parallelism / batch size (phase 3); and the **mode** (`update` probes a small declared scope → latency-bound, few hosts deep; `reconcile` probes everything → throughput-bound, whole fleet wide — *different optimizer objectives*, likely different defaults). The **audience** also shifts the objective: deployer-mode "fire, NOW" wants minimum latency-to-first-action; engineer-mode polishing wants maximum precision/insight. The cost model and these knobs are the same surface; treat them together.
