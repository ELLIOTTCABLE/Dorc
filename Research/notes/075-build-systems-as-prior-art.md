# Build systems as prior art — the closest discipline to Dorc's skip-thesis (and a vocabulary + a new perf lever)

> ⟢ 2026-06 — cross-host probe-memoization is the knob `kSTATE` (now **reopened** — persist vs stateless-recompute); the input-vs-content-addressed modes are `kELISION`; hermeticity / non-determinism-exclusion is `kVOLATILES` (welded). Grep the slug for current handling.

**The realization:** a build system *is* "performantly skip work that doesn't need redoing, derived from a dependency graph + change-detection" — which is **Dorc's MH1 thesis in another domain.** This is arguably *more* relevant to the skip core than the CM tools (Ansible/Terraform), which are the *deployment* prior art. Build systems are the *skip-decision* prior art, and they have 40 years + a clean unifying theory. Sources: **Mokhov–Mitchell–Peyton Jones "Build Systems à la Carte"** [A-mokhov-build-systems-a-la-carte-icfp-2018] (ICFP 2018, Microsoft Research, grade A — the unifying framework), Bazel/Nix/Buck2 docs (A, primary), Tweag/Aspect engineering blogs (B+).

## The à la Carte decomposition — adopt this vocabulary
Every build system factors into two orthogonal choices. Dorc has **exactly these two components** and should name them the same way (clarifies the whole design):
- **Scheduler** — *in what order* to (re)build: **topological** (static DAG, Make), **restarting** (abort+retry when a new dep is discovered, Excel), **suspending** (recurse into deps on demand, Shake). → **Dorc's apply-phase = the scheduler** (note 73's DAG walk / critical-path / list-scheduling). Dorc derives the DAG, so it's topological-or-suspending over a *derived* graph.
- **Rebuilder** — *whether* a task needs rebuilding: **dirty bit** (Make's mtime), **verifying trace** (hash deps, skip if hashes match), **constructive trace** (store the *values*, enables cloud reuse), **deep constructive trace** (store transitive deps, skip intermediates). → **Dorc's probe/skip-decision = the rebuilder.** This is the precise taxonomy for "how do you decide to skip," and Dorc's choice within it is the design crux (below).

**Two formal properties** the framework defines, which are *exactly* Dorc's soundness pair restated in build-language:
- **Minimality** — "execute a task at most once, and only if its dependencies changed." = Dorc's **elision** goal (skip everything that needn't run), and its *safety* side is elision-soundness (never skip something that DID need to run).
- **Correctness** — the build produces the right result. = Dorc's two soundnesses. The framework treats these as a *tension* (more aggressive skipping risks correctness) — the same tension Dorc navigates.

## The new perf lever this surfaced: cross-host probe memoization (constructive traces over the fleet)
**The verifying-vs-constructive distinction is the key transfer**, and it exposes a lever I hadn't named:
- **Verifying trace** (Make/Shake): store hashes of a task's deps; on next run, skip if current hashes == stored. **Local** — needs *this host's* prior-run record.
- **Constructive trace** (Bazel/cloud): store the actual *output* keyed by a hash of (command + input *contents* + env + platform); **an identical action run *anywhere* produces the same key** → "if another machine already built this with the same inputs, fetch the result instead of rebuilding." **This is what makes cloud/shared build caches work.**
- → **Dorc's fleet is precisely the cloud-build setting: H hosts, many near-identical.** If host A's probe establishes "`pkg:nginx` clean @ 1.24 under config-hash X," and host B has the *same relevant config-hash*, **B's verdict can be *reused from A* without re-probing B** — a constructive-trace / **cross-host memoization** of probe verdicts, keyed by (the relevant fact-slice's content-hash). On a fleet of 1000 identical web servers, you probe the *equivalence class* (a handful), not 1000 hosts. **This is a genuine, previously-unnamed perf lever** — fleet-size-linear probing → equivalence-class-linear. (SUSPECT it's a major win for the homogeneous-fleet case, exactly the case where note 73's "when Dorc loses" worried the probe was pure overhead — memoization *rescues* that case.)
- **The precondition is the same one build systems require (below): determinism/hermeticity** of the keyed inputs. The relevant-fact-slice must be a *complete* content-key (capture everything the verdict depends on) or the memoized verdict is unsound (an elision-soundness violation: reuse a "clean" verdict for a host that secretly differs). So the content-key must be **derived from the same dependence analysis** that computes the probe — the analysis already knows the fact-slice a verdict depends on (note 53 region/fact partition). **The memoization key falls out of the dependence graph.** Flag as a real decision (precision of the key vs reuse rate).

## Content-addressing unifies the two modes (the deepest finding)
Nix's **input-addressed vs content-addressed** distinction maps *exactly* onto Dorc's two modes (AGENTS contract matrix) and resolves a worry from the earlier conversation:
- **Input-addressed** (classic Nix): store path = hash of *inputs*. Changing a source changes the hash of *everything transitively downstream* → **rebuilds even when the output is identical** (the "comment-only change rebuilds the world" problem).
- **Content-addressed** (CA Nix): store path = hash of *output contents* → a change producing identical output (a comment, a reformat) has the same output-hash → **downstream is "cut off early," not rebuilt.**
- → **`dorc update` (git-diff scope) is input-addressed reasoning**: "the *source* changed, so re-examine this." It inherits input-addressing's flaw — a cosmetic change (comment in a role, reordered-but-equivalent commands) would re-examine downstream even if no *state* changes. **The probe is Dorc's content-addressed early-cutoff *over* the git-diff's input-addressing:** the git-diff says "*maybe* this changed" (input-addressed, cheap, over-approximate); the probe checks "did the actual *host state* need to change" (content/state-addressed, authoritative). **So the two modes aren't rivals — they're the input-addressed and content-addressed layers of one system**, and `update` = "trust input-addressing for scope, content-address (probe) within it," `reconcile` = "content-address (probe) everything." This is a clean theoretical seating of the modes, and it says: **the git-diff should *never* be the skip authority (input-addressing over-skips on cosmetic-equivalent / under-skips on cosmetic-different); the probe is — exactly the conclusion reached earlier, now with build-theory backing.**

## Early cutoff — a precision lever Dorc hasn't named
**Early cutoff**: if a task reruns but produces *the same output*, its dependents are NOT rebuilt (even though the task "ran"). → **Dorc analog:** if a mutation runs but the resulting **state-fact is unchanged** (re-applying a config-write yields a byte-identical file; `apt install` of an already-current version is a no-op), then **downstream units that depend only on that fact can still be skipped** — the mutation "ran" but "changed nothing observable," so the cutoff propagates the skip. Dorc's fact-domain (note 53) is the natural place for this: a mutation's *effect* is "establishes `fact F`"; if F's value is unchanged post-run, dependents keyed on F cut off. **This recovers skips that a naive "it ran ⇒ everything downstream must re-run" would forfeit** — and it composes with the change-signal-as-first-class-value idea (handlers/notify dissolve into this, orchestration-go-no-go). Name it; it's a real precision win at the apply/re-probe boundary.

## Hermeticity = Dorc's non-determinism exclusion (independent cross-domain validation)
Bazel/Nix caching is **correct only if builds are hermetic** — depend solely on *declared inputs*, produce *identical output for identical input*. Non-hermetic actions (system time, absolute paths, network state, `$RANDOM`) **break cache correctness** (different output-hash for "same" inputs → either spurious rebuilds or, worse, wrong cache hits). → **This is *exactly* Dorc's settled principle 3 (non-determinism excluded, not modeled; oracles canonicalize) — and the build-systems world arrived at the identical conclusion *for the identical reason*: non-determinism breaks the skip-cache.** Strong independent validation that excluding/canonicalizing non-determinism isn't a Dorc shortcut, it's a *necessary precondition for any sound skip-decision system.* The build world's "hermetic" = Dorc's "canonicalized fact"; its cache-poisoning failure mode = Dorc's false-skip (elision-soundness violation). **Borrow the framing: Dorc requires "hermetic oracles" — a check's verdict must be a pure function of declared host-state, canonicalizing volatile inputs.** (And the cross-host-memoization key above is exactly a hermeticity boundary.)

## The change-detection substrate (Buck2/DICE/Watchman/Salsa convergence)
Buck2's **DICE** (Distributed Incremental Computation Engine) is the same family as **Salsa** (rust-analyzer, note 71) and **Adapton** — a stored computation-graph, invalidate-affected-nodes-on-change, **early cutoff** built in, parallel where independent. **Watchman** (filesystem-watch) feeds change-detection *without* rescanning the tree. → Confirms the **incremental-computation-engine convergence** already noted (Salsa/Reviser/IncA, notes 53/71): the whole field independently builds the same thing. For Dorc: the analyzer's incrementality is a *solved-shape* problem (DICE/Salsa as references), and **change-detection can be event-driven (Watchman-style on the ops-repo) not full-rescan** — relevant if Dorc ever watches a repo for continuous reconcile. Reinforces note 71's "version the summary format / treat persistence as optimization" (DICE persists; Salsa doesn't — same split).

## "Build without the bytes" — don't move data only consumed downstream
Bazel's **Build-without-the-Bytes**: don't download intermediate outputs that are only consumed by *other remote actions* — keep them remote, move only what's needed. → **Dorc analog:** in the gather/verdict layering (note 70/74), don't ship probe *data* back to the controller if it's only consumed by *another probe/verdict on the same host* — compute the verdict **host-local** and return only the verdict, not the raw gathered facts. Minimizes the fan-in (note 72 observability tension). The fact-slice that's only an intermediate for a local decision never crosses the network. Reinforces "structured bounded per-host results over raw streaming."

## Net for the architecture
Build systems give Dorc: **(1) a vocabulary** (Scheduler × Rebuilder; minimality; early cutoff) to adopt directly; **(2) a new perf lever** — cross-host probe-verdict memoization (constructive traces over the fleet), keyed by a dependence-derived content-hash, turning fleet-linear probing into equivalence-class-linear and *rescuing the homogeneous-fleet "Dorc loses" case*; **(3) a theoretical seating of the two modes** as input-addressed (`update`/git-diff) vs content-addressed (probe) layers, confirming the probe (never the diff) is the skip authority; **(4) early cutoff** as an un-named apply-time precision lever (mutation ran but fact unchanged ⇒ dependents still skip); **(5) independent validation** that non-determinism-exclusion ("hermetic oracles") is a *necessary precondition* for any sound skip system, not a shortcut. Decisions to flag: the memoization-key precision (reuse-rate vs soundness), and whether the rebuilder is verifying (per-host) or constructive (cross-host) — SUSPECT **constructive/cross-host is the high-value choice** and should be designed in, since the fleet is the cloud-build setting.
