# Phase 3 — mutation deployment performance: ordering, batching, partial-failure, the under-investment trap

The user's own flag: this is "a Hard Thing that isn't really in my domain, that Ansible et al are probably already very good at, and a very easy thing to accidentally ignore in pursuit of ivory-tower 'but we can *avoid* so much work!'" That instinct is correct and is the organizing principle of this note. **Avoiding work (the analysis thesis) and doing-the-remaining-work-safely-at-scale (this phase) are different problems; the second is mature prior art and must not be hand-waved.** Sources: Terraform internals (stategraph.com, B+, human/technical), HashiCorp docs (A), Ansible docs (A), k8s docs (A), scheduling theory (Wikipedia→Graham/CPM/RCPSP primary results, B for the summary / A for the underlying theorems).

## The core realization: Dorc *derives* the DAG the others make you author — so it inherits scheduling theory, and its NP-hardness
- **Terraform** builds a dependency DAG (explicit `depends_on` + inferred from references), then **walks it in parallel up to a semaphore** (`-parallelism`, **default 10**; `internal/terraform/context.go`). The algorithm: "find all nodes whose deps are satisfied, execute in parallel up to the limit, on completion check who's now unblocked." **The critical insight (stategraph, B+):** "*If your plan contains 500 resources but only 6 are independent at a given point, parallelism=50 will not manufacture more concurrency. **Actual concurrency depends on dependencies, not the setting.**"* → **Achievable apply-parallelism = the DAG's width, and Dorc *derives* that DAG** (from imperative control-flow + fact-dependencies, round-2 / orchestration-go-no-go). So: **the analysis precision that finds independence (1B) directly sets the deployment parallelism.** Precision isn't only about skipping — it's about *unlocking concurrency at apply*. This ties phase-1 precision to phase-3 throughput.
- **Scheduling theory Dorc gets for free once it has the DAG** (and should use, not reinvent):
  - **List scheduling** (greedy: maintain a ready-list of dep-satisfied tasks, assign highest-priority to any free worker). Graham's classic bound: **(2 − 1/m)-approximation** for makespan on m workers. Good enough, standard.
  - **Critical-path prioritization** (CPM): the longest dependency chain = the **makespan lower bound under unlimited parallelism**; prioritize tasks by how much work depends on them (critical-path length). Off-critical-path tasks have slack. → Dorc should schedule derived mutations **critical-path-first**, not source-order.
  - **Graham anomalies** (surprising "wish I'd known"): **adding workers, or *relaxing* constraints, can paradoxically *increase* makespan.** → Naive "more parallelism = faster" is false; the scheduler can get *worse* with more resources. Don't expose raw parallelism as the only knob; the *schedule* matters more than the width.
  - **RCPSP** (resource-constrained project scheduling): when tasks also compete for **limited shared resources** (not just precedence), scheduling is **NP-hard**; makespan is bounded below by **max(critical-path, total-resource-demand / capacity)**. → This is exactly the **thundering-herd / shared-backend** problem (note 72) re-cast as scheduling: the apt mirror / registry / remote controller / license server are *resources* with capacity, so optimal batching is RCPSP. Dorc shouldn't solve NP-hard optimally — use **priority-rule heuristics** (resource-aware list scheduling) — but it should *know the resource constraints exist*, which it **derives** (the fact-domain partition says which mutations hit the same backend).

## The batching / blast-radius / partial-failure knobs (catalog precisely — each is a scar)
These are the mature controls. They are **SEAM/executor concerns** (orchestration-go-no-go), but the *information* to drive them is **derived by the analysis** — that's the integration seam.
- **Ansible** (docs, A):
  - **`serial: [1, 5, "100%"]`** — rolling batches: canary 1 host, then 5, then the rest. Completes each batch fully before the next.
  - **`max_fail_percentage`** — abort the play when failed-hosts exceed a threshold of the batch (blast-radius cap). With `serial`, untouched hosts stay on the old version → bounded damage.
  - **`throttle`** — cap concurrent workers for a *specific* task/block (protect a shared resource — the RCPSP capacity knob, hand-tuned).
  - **`run_once`** — a task on one host per batch (e.g. a migration).
  - **strategy `linear` (default, lock-step barrier per task) vs `free` (each host races ahead)** — the barrier-vs-no-barrier tradeoff; `free` is faster but loses the synchronization that makes rolling safe.
- **Terraform**: the semaphore-bounded DAG walk + a *transient* plan artifact ("a plan it can later execute, aborting if assumptions changed"). No native canary/rolling (it's resource-graph, not fleet-rolling).
- **Kubernetes** (docs, A): **`maxSurge` / `maxUnavailable`** (default **25%** each) bound how many pods are over/under desired during a rollout; **readiness-gated progress** (don't proceed until the new pod is *ready*, not just *started*; surge-before-terminate maintains capacity). → The **readiness-gate** is the key transferable idea: **progress only when the just-changed unit verifies healthy** — which for Dorc is *re-probe the mutated host and confirm convergence before advancing the rollout*. The probe doubles as the readiness check. This closes the loop: mutate → re-probe → if converged, advance; else halt/rollback-batch.

## The partial-failure / resumability model (the normal case, not the exception)
- Mid-rollout failure is **routine at scale**. Ansible: failed host drops from rotation; others continue (unless `any_errors_fatal`/`max_fail_percentage`); recovery = re-run (idempotence no-ops the done ones) **re-paying full check cost**, or `--limit @retry_file`. Terraform: partial state records progress; resume avoids re-work but pays the state tax.
- **Dorc's model (from note 72, restated for apply)**: re-probe finds the still-dirty set cheaply and in parallel — **resumability without a state file and without re-paying Ansible's sequential re-check.** The apply is naturally restartable: after any failure, re-probe → new minimal mutation set → continue. The two modes shape scope (`update` = declared scope; `reconcile` = all). **This makes "doing a lot of changes on a lot of hosts with interdependencies" robust by construction** — the thing the user worried we'd ignore.

## The granularity tradeoff reappears (Terraform's state-splitting lesson, transposed)
Terraform's #1 scaling lesson (note 62, now quantified, B): a **monolithic state → 30-min+ plans** (every plan refreshes/compares *all* resources) + **whole-infra blast radius**; the community fix is **splitting state by lifecycle / team / blast-radius** — which then **trades slow-plans for cross-state dependency-management complexity** (you must wire outputs between states by hand). The art is "finding the right boundaries." → **Dorc avoids the central state entirely (no split needed), but the *same granularity tension transposes onto the analysis unit*** (the per-role/per-function summary, note 71/53): **too coarse** a unit = re-analyze too much per diff + coarse skip granularity (Terraform's slow monolith); **too fine** = summary-composition overhead + cross-unit dependency bookkeeping (Terraform's split-state complexity). The summary boundary IS Dorc's "state-split boundary" — and because Dorc *derives* cross-unit dependencies (the dependence graph) instead of making you hand-wire them, it can pick **finer granularity without paying Terraform's manual-wiring tax.** GUESS: align the unit boundary with the natural role/function structure (the embarrassingly-modular shape) and let derived deps handle cross-unit — getting fine-grained skip *and* automatic dependency management, the thing Terraform forces you to choose between.

## The under-investment trap, named (the user's central phase-3 fear)
The ivory-tower failure mode: pour all effort into the analysis (avoid work) and ship an executor that's a naive parallel-for-loop, so that the moment a real fleet has **ordering constraints, shared-resource limits, partial failures, and blast-radius requirements**, Dorc is *worse* than Ansible at the actual deployment — because Ansible spent a decade on `serial`/`throttle`/`max_fail_percentage`/strategies and Dorc spent it on lattices. **Mitigation (architectural, decide now):**
1. The **executor seam must carry the rich knobs from day 1's design** (even if v1 implements a subset): rolling batches, resource-throttle, blast-radius cap, readiness-gated progress, restartability. These are *reserved hooks* in orchestration-go-no-go — keep them reserved, don't let "we derive the DAG" lull us into a dumb for-loop.
2. The **differentiator is that Dorc *derives* the inputs** these knobs need (the DAG, the independence width, the shared-resource contention, the still-dirty set) that Ansible/Terraform make you hand-author. So Dorc can **auto-tune** what they hand-tune: derive batch boundaries from the dependence graph, derive throttle groups from fact-domain contention, derive the canary set from blast-radius (which hosts a change touches). That's the phase-3 value proposition — *not* "we avoid the work," but "we schedule the unavoidable work better because we computed its structure."
3. **Don't reach for optimality** (RCPSP is NP-hard) — heuristics (critical-path list scheduling + resource-aware priority) are the prior-art-blessed answer; "good schedule fast" beats "optimal schedule never."

## Footgun list (phase-3 "wish I'd known")
1. **A naive parallel-for executor** → loses to Ansible the instant real ordering/resource/failure constraints appear. Reserve the rich executor knobs in the design now.
2. **Exposing raw parallelism as the only knob** → Graham anomalies (more workers can be *slower*); the *schedule* (critical-path-first, resource-aware) matters more than the width.
3. **Ignoring shared-resource constraints in scheduling** → RCPSP capacity violations = the thundering herd at apply-time. Derive contention; resource-aware list-schedule; throttle.
4. **No readiness gate between batches** → cascading bad rollout. Use the re-probe as the k8s-style readiness check: mutate → re-probe-converged → advance.
5. **Treating partial failure as exceptional** → it's the norm. Make apply restartable via re-probe (free, given the architecture); never require a central state file to resume.
6. **Source-order execution** → ignores the derived critical path; schedule by dependency structure, not script line order.
7. **Letting "we avoid work" justify under-building the executor** → the central trap. The executor is SEAM, but a *first-class designed* seam, not an afterthought.
