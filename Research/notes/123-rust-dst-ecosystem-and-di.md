## Findings

- f14. **(Q1) Battle-tested users exist, with one clear flagship.** RisingWave (cloud distributed SQL, runs continuous end-to-end DST via its own madsim — the anchor) [A-risingwave-deterministic-sim-2023]; S2 (streaming-storage startup, DST on *every PR/commit* + thousands of nightly trials, found 17 notable bugs) [B-s2-dst-async-rust-2025]; NautilusTrader (OSS trading engine, turmoil + a madsim-based framework); Polar Signals (new Rust object-storage DB, state-machine DST) [B-polarsignals-dst-state-machines-2025]; sled (Tyler Neely — the origin of the Rust pattern) [A-sled-simulation-guide-2020]; `penberg/hiisi` (Pekka Enberg's TigerBeetle-style libSQL PoC, archived). +SURE the technique is in real production use in Rust, not just toy repos.
- f15. **(Q2) Libraries that give it "for free", by altitude.** Full deterministic *runtime*: **madsim** (drop-in tokio replacement) [A-madsim-deterministic-simulator-2025]. Network/filesystem *sim* on real tokio: **turmoil** (tokio-team) [B-turmoil-tokio-2023]. Shared-memory *concurrency-schedule* testing (a different axis): **shuttle** (AWS, randomized PCT scheduler) [B-shuttle-concurrency-testing-2024] and **loom** (exhaustive). The provider-trait *pattern* as a small lib: **moonpool** [C-moonpool-provider-pattern-2024]. +SURE.
- f16. **(hypothesis CONFIRMED) The compile-time swap is the dominant mechanism and is genuinely zero-cost in production.** madsim ships `madsim-tokio`/`madsim-tonic`/`madsim-etcd-client`/`madsim-rdkafka`/`madsim-aws-sdk-s3` as drop-in deps: "When built normally, these crates are identical to the original ones"; under `RUSTFLAGS="--cfg madsim"` they become the deterministic simulator. So prod build = real tokio, zero overhead; sim build = swapped. [A-madsim-deterministic-simulator-2025] +SURE.
- f17. **But it is NOT proc-macro source-rewriting — and that matters for the hypothesis.** The swap is three mechanisms, none of which rewrite your call-sites: (a) **cfg-conditional shim crates** (`[target.'cfg(madsim)'.dependencies]`); (b) Cargo **`[patch.crates-io]`** to replace transitive deps; (c) **libc-symbol overrides** of `clock_gettime`/`getrandom`/`gettimeofday`/`sysconf` so nondeterminism cannot leak through `std` or third-party libs. [A-risingwave-deterministic-sim-2023] +SURE. Proc-macro rewriting of time/IO calls is not how the Rust ecosystem does it (binary/IL rewriting is the Antithesis-hypervisor / .NET-Coyote approach, not idiomatic Rust).
- f18. **Three distinct DST architectures in Rust (the design space).** ~SUSPECT on which fits Dorc; all three are zero-cost in prod:
   1. **Runtime-replacement / cfg-swap (madsim):** retrofit-friendly, minimal code change; *but* locked to madsim's shimmed deps, and any unshimmed dep leaks nondeterminism. Best for *existing* codebases (Polar Signals and S2 both say so).
   2. **Provider-trait DI (moonpool, redis-rust):** code is generic over `TimeProvider`/`NetworkProvider`/`RandomProvider`/… — monomorphized, zero-cost, explicit, cross-platform; you build the simulator. [C-moonpool-provider-pattern-2024]
   3. **State-machine + message-bus (sled, Polar Signals):** core logic = single-threaded state machines with **no `async`** (compile-time guarantee of no runaway futures), dependencies modelled as *messages* not injected objects, and **all four ingredients (time/randomness/scheduling/fault-injection) centralized in the message bus**. Max control, best for greenfield, biggest architectural commitment. [A-sled-simulation-guide-2020][B-polarsignals-dst-state-machines-2025]
- f19. **The cross-platform determinism trap is real and load-bearing (`axis-platform`).** S2 got single-thread + tokio paused-clock + turmoil and *still* leaked nondeterminism — CI failures unreproducible on Macs — from third-party deps, randomized `HashMap`, HTTP timestamps; only fully fixed with madsim-style **libc overrides** (their `mad-turmoil`). [B-s2-dst-async-rust-2025] +SURE. The libc-override fix is itself OS-specific (Linux/Mac code paths), so DST sits awkwardly with the macOS/Windows dev loop — exactly the `axis-platform` tension.
- f20. **Everyone adds an "is it actually deterministic?" guard:** rerun the same seed and panic on any divergence (madsim's `MADSIM_TEST_CHECK_DETERMINISM=1`) or diff TRACE-level logs across two runs of a seed (S2's CI meta-test, "down to the last bytes on the wire"). [B-s2-dst-async-rust-2025] +SURE — this is the practical half of the Eaton/Will-Wilson "is my DST working" worry (it at least proves determinism; coverage is the harder, unsolved half).
- f21. **Dorc-fit nuance (−GUESS).** Dorc's "IO" is *spawning sh/ssh on remote hosts*, not (only) tokio sockets — so madsim's tokio-shim would cover Dorc's *internal* async/networking but **not** the leaf-execution seam (process-spawn + SSH). The `dorc_exec` chokepoint (`notes/122`) needs provider-trait or state-machine treatment regardless of whether madsim is also used for any internal tokio. So the relevant prior art for Dorc's *own* seam is sled/Polar-Signals/moonpool more than madsim-the-tokio-shim.
- f22. **Rust's DST maturity is itself a (minor) input to the language choice.** madsim/turmoil/shuttle/loom + the state-machine pattern + multiple production users mean a greenfield Rust correctness-critical network tool has real, trodden paths to DST. ~SUSPECT (language decision is separate and out of scope here).
- f23. **(human Q) The transitive-dependency problem is the real wall; Rust is "usually screwed" but with two levers that beat naive DI, plus an architecture that sidesteps it.** Pure DI only reaches *your* code — a transitive dep that calls `SystemTime::now`/`getrandom`, seeds a `HashMap`, or spawns a thread leaks nondeterminism you cannot inject around. Rust's tractability comes from intercepting at a **narrow waist** instead of per-dep: (a) **libc-symbol override** of `clock_gettime`/`getrandom`/`gettimeofday`/`sysconf` catches anything bottoming out in syscalls — yours, std, or deps — in one place [A-risingwave-deterministic-sim-2023][B-s2-dst-async-rust-2025]; (b) Cargo **`[patch.crates-io]`** force-swaps the named offenders that escape the libc net — madsim ships forks of `quanta`/`getrandom`/`tokio-stream`/`tokio-postgres` precisely because they read the TSC/vDSO or carry internal RNG [A-madsim-deterministic-simulator-2025]. Residue you must still hunt: vDSO/TSC clock reads that bypass libc, randomized `HashMap` `RandomState`, thread-spawning deps, entropy captured at static-init — the rerun-seed guard (f20) *detects* a leak but not *which* dep; and the libc trick is OS-specific (Mac `CCRandomGenerateBytes`, Windows needs its own) → `axis-platform`. **Approach #3 (state-machine) largely sidesteps the whole problem**: a kernel with no IO/time/rand deps has no transitive nondeterminism to police — you fight it only at the thin driver edge. **Antithesis "buys out" the problem** by intercepting *below* libc at the hypervisor, where no dep can escape (still must mock external systems). **Dorc-specific (−GUESS→~SUSPECT):** the analyzer kernel is near-pure (few nondeterministic deps), and Dorc's most important "dependency" — the remote host over ssh — is not a crate to `[patch]` but THE seam you mock at `dorc_exec` by design (f21); so Dorc's deterministic kernel can be large and its nondeterministic edges few and explicit, which is *more* tractable than a general DB, not less.

> [A-risingwave-deterministic-sim-2023]:§The-Madsim-Project / §Challenges (relevance: +1:SURE)
> Madsim intercepts some critical std APIs by overloading libc functions, such as gettimeofday, clock_gettime, getrandom, and sysconf, to provide deterministic time, random numbers, and system configuration in the simulation environment. This prevents uncertainty leaks caused by accessing the real world through std and third-party libraries.
> Third-party libraries with internal randomness cannot be used, and all code on all nodes must be executed on a single thread. This can be achieved by designing state machines and event loops or by introducing coroutines.
> The queue is designed to be first-in-random-out (FIRO) to explore potential concurrency issues.

> [A-sled-simulation-guide-2020]:§recipe (relevance: +1:SURE)
> write your code in a way that can be deterministically tested on top of a simulator. This also ensures you're properly applying the dependency inversion principle.
> fn receive(msg, at) -> [(msg, destination)] … fn tick(at) -> [(msg, destination)]
> stuff all messages / events in the system into a priority queue keyed on next delivery time … for each outgoing message … deterministically assign a "delivery time" to the message (or drop it).

> [B-polarsignals-dst-state-machines-2025]:§architecture (relevance: +1:SURE)

> The state machine interface does not include the `async` keyword, which ensures at compile time that there are no runaway futures spawned inside state machines.
> this failure injection needs to be implemented only once in the message bus to apply to all state machines (including all dependencies).

> [B-s2-dst-async-rust-2025]:§Making-it-deterministic / §Blending / §Takeaways (relevance: +1:SURE)
