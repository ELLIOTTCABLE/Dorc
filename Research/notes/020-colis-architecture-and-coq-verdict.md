# CoLiS architecture, corpus evidence, and the "is Coq justified?" verdict

## The CoLiS toolchain architecture (TACAS 2020) [A-colis-installation-scenarios-tacas-2020] — what to crib, what to drop
Pipeline: `Package + Scenario → Scenario Player → Front-end(parse + translate) → Back-end(symbolic engine) → Diagnosis`.
- **Front-end** = Morbig (shell → concrete syntax tree) **then a translation into `CoLiS`**, a clean intermediate language with formally-defined semantics.
- **Back-end** = a symbolic-execution engine that computes, per outcome, a symbolic **input→output filesystem relation** encoded as **feature-tree constraints**; the engine is **generic over UNIX-command specifications, which are passed in as parameters.**

SURE — the load-bearing architectural lesson, and it maps almost 1:1 onto Dorc:
- **CoLiS already separates "the engine" from "the per-command knowledge."** The symbolic engine is generic; each UNIX command's effect is a *plug-in specification*. This is exactly Dorc's *analyzer-engine + pluggable community oracles* split — and CoLiS is an existence proof that this separation scales to ~28k scripts and ~hundreds of commands. **Crib the separation.**
- **They run analyses over a clean IR (`CoLiS`), never over raw shell.** Raw shell is parsed to a CST, then *translated* into the analyzable language; anything untranslatable is rejected up front. Dorc's "strict superset" is the *inverse-direction* version of the same idea (we define the clean language and let users migrate in, rather than translating arbitrary shell in and rejecting 23%) — but the principle "analyze an IR with defined semantics, not raw shell" is shared. **Crib the IR-as-analysis-target.**
- **What to DROP:** the symbolic-execution + feature-tree-FS-relation machinery. That heavyweight relational-FS reasoning exists to *soundly over-approximate* the filesystem transformation — i.e. it serves the soundness goal we've rejected. Dorc replaces the per-command *formal FS-relation spec* with a per-command *executable check-oracle shipped to the host* (probe real state, don't symbolically model it). We keep the engine/oracle split; we swap symbolic-reason for host-probe.

## Corpus-mining findings — empirical backing for "even a dumb analyzer wins"
SURE, and these directly de-risk MH1 (the planning-log's flatness/simplicity bet, now with numbers). Debian maintainer-script corpus = 28,814 scripts, 12,592 packages, 442k SLOC, **avg 15 lines, max 1138**:
- **99.9% parse** (Morbig). Translation to the analyzable IL succeeds for **77%**; the 23% failures are concentrated in **globs, parameter expansion (`${x:-y}`), and advanced redirections** — a *small, enumerable* set of features. → for Dorc these become defined-or-`unsafe` decisions, not open-ended parsing research.
- **Variables are constants**: only 3,008/28,814 scripts (~10%) contain a variable whose value actually changes. → static expansion handles ~90% of scripts.
- **Zero recursive functions** in the entire corpus.
- **`while` loops are an idiom**: 2,300 scripts use `while`; **93% are the `dpkg -L | while read` pattern**, mechanically convertible to `for`. → control-flow diversity in real install scripts is tiny.
- Redirections are almost always "discard stdout / merge stderr."
GUESS: this is the strongest external evidence that Dorc's thesis is sound — real provisioning shell is short, constant-heavy, shallow, and idiom-bound. The CFG/effect analysis needs to be *good*, not *complete*, to skip the bulk. (Caveat: this corpus is Debian *maintainer* scripts, a narrower/cleaner population than the user's actual target of "janky app install from a non-ops dev." `bash-in-the-wild` is the broader-population check — read next.)

## Hard-won lessons CoLiS reports (all confirm planning-log decisions)
- **Idempotency was nearly impossible to even define** for real Debian policy ("if the first call aborted halfway, the second should merely do what was left undone"). They gave up on a real formalization and modeled only a *rough under-approximation comparing exit codes.* → Strong confirmation of the planning-log decision to NOT chase idempotency-as-proof, and that convergence-from-arbitrary-partial-state is genuinely ill-defined, not just hard.
- **"Identifying bugs always requires human examination."** Their tool surfaces candidates; humans adjudicate. → matches Dorc's "surface candidates, best-effort" posture; an automated skip/flag is an *advisory*, not a proof.
- They consciously **don't model permissions/timestamps/symlink-resolution/concurrency** — accepting both over- and under-approximation. → confirms "bound the model; exclude what you can't cheaply nail" (our exclude-non-determinism principle).

## The "is a proof assistant (Coq/Isabelle/Why3) justified?" verdict
Evidence from VSTTE 2017 (the CoLiS-language verified interpreter) [A-verified-interpreter-shell-vstte-2017] + the TACAS trust boundaries:
- The CoLiS group — maximally formal-methods-capable — verified **only the interpreter/semantics core of a *clean, self-designed* intermediate language**, and did it in **Why3 + automated SMT provers (Alt-Ergo/Z3/E/CVC)**, *deliberately avoiding* interactive assistants (Coq/Isabelle) as too heavy ("formalising semantics is most often done in Coq/Isabelle; we show automated provers suffice"). Even so, the *completeness* proof needed a **novel "ghost skeleton" technique** (abstracting the proof tree to a decreasing algebraic value) — research-grade effort, 207 subgoals, for ONE interpreter of a SIMPLIFIED language.
- Crucially, they **did NOT and COULD NOT formally verify the two parts Dorc cares most about**: (i) the **parser** (POSIX spec is informal → "impossible to prove correct; we don't even claim absence of bugs"), and (ii) the **shell→IL translation** (no formal shell semantics existed). Both are *trusted via review + differential testing against dash/bash.*
- The formal verification bought them **soundness of a symbolic over-approximation** — exactly the goal Dorc rejected.

SURE verdict for the Phase-1 plan: **No to Coq/Isabelle. No, even, to CoLiS-style Why3+SMT as a project methodology.** Rationale: Dorc's domain (best-effort over nearly-arbitrary shell) is precisely the regime where CoLiS *itself* fell back to trust+differential-testing; the parts amenable to proof (a clean IL's interpreter) sit behind a trusted translation that can't be proven, so end-to-end soundness is unattainable regardless — proving the middle doesn't make the system sound. The investment (PhD-scale) serves the 5% tail the user disclaimed.
- **What replaces it (the calibration mechanism):** the CoLiS *parser* trust model, which is also literally the user's "test-container toolkit" idea — (i) embed the spec/grammar verbatim where one exists, (ii) **differential testing** of our analysis against real shells / real host outcomes, (iii) property-based testing (parse∘pretty-print idempotence; check-oracle vs. actually-running-the-mutate). This is engineering-grade calibration, not proof.
- **Narrow caveat I'll still flag in the plan (GUESS):** *selective, targeted* lightweight verification of one or two algorithmically-subtle kernels (e.g. the effect-lattice join/monotonicity, or the check-projection's "⊥ ⇒ provably-no-mutation" invariant) *could* be worth a Why3/property-test harness — not as methodology, but as a unit-level confidence tool for the places a bug is silent-and-dangerous (the probe-soundness inert-classification kernel, and the elision skip-only-on-proof kernel — see AGENTS §1). Probably v2+; note, don't adopt.

## Decision-procedure / formalism note (for the architecture section)
CoLiS's "feature-tree logic" (trees with filenames on edges, kind {dir,reg,symlink} on leaves; constraints feature/kind/similarity/absence/maybe; first-order theory **decidable** via quantifier elimination — Jeannerod & Treinen) is the formal heart of their FS modeling. SUSPECT we do **not** want this: it's the apparatus for *reasoning about* FS transformations symbolically. Dorc *probes* real FS state via oracles instead. Keep the feature-tree work as a reference for *the canonical-fact ontology* (Tier-B), not as an engine component.
