### domain-1 · parse-error handling/recovery + provenance (turn-1, 4 sources read)

Highest-signal / most-crosscutting first. Markers per CLAUDE.md (+SURE/~SUSPECT/-GUESS/--WONDER).

- f1. **The central shape: outcome goes from a sum `Result | Error` to a product `Result × [Error]`.** Merlin's recovery "changes the possible outcome of the frontend from either a typed AST *or* an exception, to a typed AST *and* a list of exceptions" — *always* produce a best-effort result AND an accumulated diagnostic-list; downstream always has something to consume. +SURE this is the exact "continue-at-all-costs then report" the prompt described, and it is **crosscutting**: every Dorc stage (parser→analyzer→planner→orchestrator) should return `(best-effort-result, accumulated-diagnostics)`, never throw-and-die mid-pipeline. [A-bour-merlin-2018]
- f2. **Fail-fast *detection* ≠ fail-fast *reporting*.** LR "guarantees an error is detected as early as possible … the recovery algorithm makes use of this property by *not* trying to backtrack: all input parsed is kept" — detect at the earliest impossible token, then keep everything-so-far and fill *forward*. +SURE this reconciles the prompt's "I'll pursue fail-fast" with "best reporting looks like continue-recovery": fail-fast is the *detection discipline*, not a stop-the-world. [A-bour-merlin-2018]
- f3. **Error nodes are provenance-carrying placeholders, not absence-of-data.** Keep ill-typed/unparseable nodes *specifically to preserve source positions* so downstream position-keyed features still resolve; synthesize a "fake" node carrying the context-expected type + the salvaged partial children. +SURE **crosscutting**: a failed leaf in Dorc's CFG/effect graph must still carry its `sh`-span + partial facts, not vanish. Mirrored independently in Clang `RecoveryExpr` and matklad's `ErrorTree`/`ErrorToken`. [A-bour-merlin-2018] [B-matklad-resilient-ll-2023]
- f4. **Tag synthesized/inferred data distinctly from user-supplied data, at the value level, so the *consumer* adjudicates trust.** grmtools hands actions `Result<Lexeme,Lexeme>` (Ok = real user token, Err = recovery-inserted); Merlin only synthesizes values that grammar annotations license. ~SUSPECT this is the closest prior-art analogue to Dorc's must-have **we-inferred-this vs user-wrote-this provenance bit** on every fact — and the lesson is it rides *on the data itself*, not a side-table. [B-tratt-error-recovery-2020] [A-bour-merlin-2018]
- f5. **Where does error-metadata live? Inline-in-the-tree vs side-effect-of-the-process is an explicit, load-bearing choice.** matklad: "errors as properties of the syntax tree itself … or … a side-effect of the parsing process (that way, trees constructed manually during refactors won't carry error messages)." +SURE this *is* Dorc's `kOOB` (in-band ↔ sidecar) recurring in the parser; the refactor-tree argument is a real reason to favour the side-channel for *derived* diagnostics. [B-matklad-resilient-ll-2023]
- f6. **The diagnostic catalog is a separate declarative artifact, kept *complete vs. the engine* by a mechanical coverage check.** Menhir/Pottier: grammar and the (state→message) collection are two declarative specs; a reachability algorithm enumerates *every* error state so the catalog can be proven complete and *stay* complete as the grammar evolves. +SURE design-affecting: Dorc's analysis-failure messages can be a separate catalog with a completeness gate firing when the analyzer's "error states" change — decouples message-quality work from engine work. [A-pottier-reachability-2016]
- f7. **Static-state info is cheap; dynamic-stack/context info is expensive to summarize — and the split forces over/under-approximation of "what's valid next."** Good messages "recall the past and explain the valid futures," but from the static state alone the valid-futures set is sometimes unavoidably over- or under-approximated (Menhir adds `%on_error_reduce` + the "If this expression is complete, then…" hedge). ~SUSPECT this maps cleanly onto Dorc: facts from the static CFG (cheap) vs facts only the cross-network probe knows (expensive), and the over/under-approximation duality is literally `kFAIL` (probe under-approximates, apply over-approximates). [A-pottier-reachability-2016]
- f8. **Best-effort completion is cost-model-guided, and "never do this" is spelled as infinite cost.** Merlin annotates each shift/reduce with a `[@cost n]`; completion = minimum-combined-cost search; an action that can fail is given *infinite* cost to exclude it. +SURE maps to Dorc's cost-model (`Q-COSTVEC`) and gives a clean idiom for steering inference away from unsound guesses — the dangerous completion isn't special-cased, it's just priced out. [A-bour-merlin-2018]
- f9. **One diagnostic spine, not per-phase silos: convert lexer errors into parser errors so a single recovery/reporting path serves both.** Tratt: a catch-all `. "ERROR"` lex rule turns lex failures into ordinary parse errors → "lexing and parsing errors are now presented uniformly." Merlin similarly refuses to special-case lexer recovery. +SURE **crosscutting**: Dorc should funnel parser/analyzer/planner/orchestrator faults into one diagnostic representation, not four. [B-tratt-error-recovery-2020] [A-bour-merlin-2018]
- f10. **Suppress spurious cascades by promoting a *local* repair to a *regional* one.** CPCT+ computes the complete min-cost repair set, then temporarily applies each and keeps only those that parse furthest ahead (bounded ~250 tokens) — "reduces spurious syntax errors *and* reduces the quantity of low-quality repair sequences we present." +SURE the failure mode to beat (javac panic-mode → users learn to ignore everything after the first error) is a *reporting-quality* bar, and the fix is cheap (~20 LoC) once you keep the full candidate set. [B-tratt-error-recovery-2020]
- f11. **Exposing the engine's state as a first-class *inspectable, purely-functional* value paid off in unplanned reuse.** Merlin's `checkpoint`/`step` API (user feeds tokens, control inverted; states cached by location for incremental restart) enabled three uses its authors didn't plan: state-indexed error messages, a pure "lexer hack," and hand-rolled local-GLR. +SURE **crosscutting**: a clean inspectable "where we are / what we know" representation is a reuse multiplier — worth designing Dorc's analysis-state to be one. Pure states also *are* the incremental-recompute mechanism (restart from last checkpoint before the edit; requires pure semantic actions ≈ `kVOLATILES` hermeticity). [A-bour-merlin-2018]
- f12. **A long-running analyzer must invalidate against on-disk ground truth.** Merlin "regularly checks whether a cached compiled interface changed on the filesystem, or if a new file … could shadow a module, and recreates a coherent typing environment." ~SUSPECT directly relevant to `kSTATE`: an operator-node Dorc process that caches host facts/verdicts must re-validate against host/disk reality, not trust its own cache — the stale-cache hazard is first-party here, not hypothetical. [A-bour-merlin-2018]
- f13. **The perspective shift recovery forces: "is this correct *now*?" → "would this be correct in a *future extension* of the environment?"** Merlin treats a missing record label as "should be added later," not fatal. --WONDER this open-world framing is the same posture as Dorc's `kDEPS-accept-partial` (knowledge is non-total, fill best-effort) and may be a citable theoretical anchor (Merlin links it to Omar et al.'s editor-inspired type systems / Hazelnut). [A-bour-merlin-2018]

### domain-1 · provenance-threading: carrying location/origin/trust through every node (turn-1, 3 sources read)

- f14. **Provenance = a cheap interned handle threaded through every node, resolved lazily against a side-map.** rustc's `Span` is attached to most HIR/MIR constructs but is just a handle; the full snippet is fetched from a `SourceMap` only at report time. +SURE the implementation pattern for Dorc: every CFG/effect-graph fact carries a compact span-id, never the `sh` text; resolve to the snippet only when reporting — keeps the hot analysis path small (the memory-wall / `kFACTS`/`kPRECISION` concern). [B-rustc-diagnostics-2024]
- f15. **The span is "spatially aware" — it points at the code, so the message can be terse.** rustc: a primary span "should have enough text … if it were the only thing being displayed (IDE) it would still make sense … more succinct than the error message." +SURE fits Dorc's dogfood-in-sh aesthetic: lean on pointing-at-the-sh (the span carries context) over prose. [B-rustc-diagnostics-2024]
- f16. **A proposed *fix* carries a machine-readable confidence (Applicability), separate from the message.** rustc: `MachineApplicable → MaybeIncorrect` decides whether a tool may auto-apply. +SURE this is provenance+confidence on a *suggested change*, mapping onto (a) Dorc's plan/apply, where a proposed `sh`-edit needs an "auto-apply-safe?" tag, and (b) the CLAUDE.md certainty markers. Crosscutting: every Dorc hint ("this looks like a guard — write an oracle") should carry an applicability. [B-rustc-diagnostics-2024]
- f17. **Track where in the *tool itself* a diagnostic was created (meta-provenance).** rustc `-Z track-diagnostics` prints the compiler source-location that emitted each error; the `Diagnostic` trait "enforces a separation between diagnostic emitting logic and the main code paths." ~SUSPECT for a best-effort engine that *will* be wrong often, attributing a report to which pass/oracle/rule produced it is the core debugging lever — and it re-argues f6: emit diagnostics-as-data, decoupled from the logic that triggers them. [B-rustc-diagnostics-2024]
- f18. **Provenance packs into one integer — file+line+col+range — via interval→map tables, with an ad-hoc side-table escape for rich cases.** GCC `location_t`: ordinary maps slice the int space into (file, line, col-bits=7, range-bits=5); macro maps sit near 2^31; AD-HOC locations above 2^31 carry whatever won't fit. +SURE the canonical compact-provenance encoding — directly relevant to carrying provenance on *millions* of Dorc facts cheaply, with an escape hatch for the few needing full history. [B-gcc-libcpp-location-2024]
- f19. **Provenance survives a source-to-source transform by *dual-encoding* origin: a macro-expanded token records BOTH use-site and definition-site in one location.** GCC macro maps give each expanded token an x-location (use/expansion point) and y-location (macro-definition point); a token straight from the macro body has x==y; each ordinary map also records "included from location L" (an arbitrary-depth `#include` chain). +SURE **the key crosscutting insight**: Dorc's probe-compilation (sanitize + parallelize + strip) *is* a source-to-source transform, so every shipped-probe line must back-point to the originating user/oracle `sh` line — exactly GCC's macro dual-location. This is what makes a probe failure on a remote host attributable to "your oracle line 12, lifted into the probe for nginx," and it pins down the `kFIDELITY` 1:1 seam as a provenance requirement, not just a debug mode. [B-gcc-libcpp-location-2024]
- f20. **Metadata is detachable and transplantable — that is *how* provenance crosses a transformation.** Racket syntax object = datum + {lexical-info, source-location, syntax-properties, tamper-status}; "the lexical information … is independent of the other half, and can be copied to a new syntax object in combination with an arbitrary other value." +SURE a transform constructs new output and *grafts* the input's provenance onto it (what macros do); for Dorc, when the analyzer rewrites/lifts `sh` it should graft origin-provenance onto each produced node, not regenerate positions. Three riders worth stealing: arbitrary `syntax-properties` (in-band sidecar metadata that survives expansion ≈ `kOOB` but co-located with the node), a `tamper-status` *trust* bit (security provenance — cross-links the security round's `kTRUST`), and the `label` phase-level = a tracked reference edge that explicitly carries NO execution dependency (Dorc wants provenance-edges distinct from control-flow-edges). [A-racket-syntax-model-2024]
- f21. **(synthesis across f1–f20) The "central/coherent language" the prompt asked about has a recognizable shape.** Across parsing (matklad/Merlin), repair (Tratt/Pottier), and provenance (rustc/GCC/Racket) one data-structure recurs: a node = `(best-effort value | error-placeholder)` + a compact origin-handle + a confidence/trust tag + optional grafted-through-transform provenance; and the process = always produce `(result × accumulated-diagnostics)`, never throw; with the message/catalog decoupled from the engine and kept complete by a coverage check. ~SUSPECT this is the shared structure to carry across parser→analyzer→planner→orchestrator. The cross-network boundary (domain-3, next) is precisely where the accumulated-diagnostics *list* and the origin-handle-resolves-against-a-side-map both become **distributed** — the part the prompt flags as scary and that this round hasn't yet touched.

### domain-1 · deepening: error-node design, cascade-suppression, relative-provenance, history-recovery (turn-2, 4 sources read)

- f22. **Use a dedicated error/poison node-kind, not an "invalid" flag on every node.** Clang's RFC rejects "emit the CallExpr anyway + an invalid flag" because it "weakens many invariants in many different node types … subtle and error-prone"; instead one generic `RecoveryExpr` preserves {attempted-kind, still-valid children, best-guess type}. +SURE crosscutting — Dorc's failed-analysis result should be one explicit ⊤/error node-kind (matklad `ErrorTree`, Merlin fake-node, Clang `RecoveryExpr` all converge), keeping every other node-type's invariants intact. [B-clang-recoveryexpr-2019]
- f23. **Cascade-suppression = give the poison node a type that absorbs all downstream checks.** Clang gives a `RecoveryExpr` a synthetic "unknown type not convertible to anything" so it "won't pick an unwanted overload," and "does not report most errors on dependent expressions, so we get rid of bogus errors for free." +SURE Dorc analog: a ⊤-poisoned fact propagates as "unknown — conclude nothing downstream," silencing follow-on diagnostics in the *lattice* rather than by ranking (the principled twin of f10). [B-clang-recoveryexpr-2019]
- f24. **Different consumers need different visibility into error nodes.** Clang: "a correctness check may want to ignore `RecoveryExpr` nodes whereas a refactoring check may actually want to look through the node"; visitors get a see-through-or-not switch. ~SUSPECT for Dorc the probe-planner (wants only solid facts) and the reporter/linter (wants the failures) traverse the same graph with opposite error-node policies — make visibility a traversal parameter, don't hardcode. [B-clang-recoveryexpr-2019]
- f25. **Store provenance RELATIVE (offset/width), derive absolute position lazily in a thin contextual wrapper — that is what makes provenance-bearing nodes reusable across edits and shareable as a DAG.** Roslyn green nodes store {kind, width, children, diagnostics} with NO absolute position and NO parent; the red layer computes Span+Parent on demand. Identical subtrees (`()`) share one cached green node (DAG; 55% cache hit), and edits reuse ~99.99% of green nodes because a node "at position 10,000 can be reused at 10,050 — only the red wrapper's position changes." +SURE the production answer to GCC's compact-location (f18) plus the incremental angle: for `dorc try`, store facts/provenance relative-to-enclosing-construct so surrounding-`sh` shifts don't invalidate unchanged subtrees, and let diagnostics ride the immutable shareable core. [B-roslyn-redgreen-2024]
- f26. **History-sensitive recovery: with edit-history + per-node change tracking, the error *message* can BE the diff — the rejected change — not an authored string.** Wagner's *non-correcting* recovery refuses to incorporate the invalid edit (no repair-guessing; "spurious repairs never arise") and presents the error as "the difference in the content of the isolated region between V2 and V0," correlating the detection point back to the actual causing user edit even across distance/intervening errors. +SURE **the `dorc try` diff-mode prior-art**: a failed partial-apply's diagnostic is "the specific `sh` change you made that I couldn't incorporate," computed from provenance+history, needing NO message catalog for the change-rejection case (catalog still needed for fresh material — complements Pottier f6). Error *isolation* via dependency structure lets unaffected valid edits still apply (`kELISION`). [A-wagner-history-recovery-1997]
- f27. **Per-node local-vs-nested change bits are the incremental-reuse + change-attribution substrate.** Wagner: every node reports `local` (directly edited) and `nested` (on the root→modified-descendant path) changes, versioned; three versions (Reference=last-consistent, Previous=input, Current=building) bound what's reusable. ~SUSPECT directly the bookkeeping `dorc try` needs: mark which facts/subtrees the edit touched (local) vs lie on a path to a touched one (nested), recompute only those — the change-attribution that makes diff-mode sound. [A-wagner-history-recovery-1997]
- f28. **(corroboration, quantified) the cascade failure mode and its fix are measured; hand-rolled recovery is uneconomical.** Diekmann/Tratt on 200k real Java errors: ranked complete-repair-set cuts reported error-locations 981,628→435,812 (~55%) at 98.37%@0.5s; the worst-ranking variant reports +31.93% errors; Eclipse's hand-written Java recovery is 5KLoC (~a whole parser) vs CPCT+'s <500 lines; panic mode's root sin is that it "implicitly deletes input from before the error location." +SURE corroborates f2/f10 with numbers and supplies a calibration-harness template (real-error corpus + per-file time-budget) for `kVERIFY`. [A-diekmann-cpct-2020]

### ‼ DIRECTION (human, turn-2): provenance/errors are N-tier, *forking*, and span static→dynamic→distributed→dependency — domain-1 only covers ONE tier-hop

The human flagged that everything above models at most a *two*-tier locus (Menhir/GCC: source ↔ one transform). Dorc's real structure is **more layered AND tighter-integrated**. Worked example, decomposed:

> a parse-*warn* (that did **not** error out the transformation) → gets transformed **differently per-host** → shipped to separate hosts → each host appends a **different** runtime error from what-actually-happened during *that* probe → finally surfaced to the user as an annotation on a *line-of-mutation that **depends on** that probe*.

- f29. **Provenance is a derivation DAG over ≥4 orthogonal dimensions, not a chain.** ~SUSPECT the dimensions:
  - **transform (static, composable >2):** user-`sh` line → oracle-lift → sanitized/probe-compiled line. GCC macro dual-location (f19) is *one* hop; here it composes to depth N (a source-map-of-a-source-map).
  - **fork (static→distribution):** the transform is **per-host**, so one source line → **N different** mutated-probe lines. Provenance *fans out*; the same origin has N distinct downstream loci. None of the domain-1 prior-art forks.
  - **distribution:** every downstream locus is **host-qualified** (which host, which probe-run/session).
  - **runtime (dynamic, remote):** what actually happened during probe execution on host H — the host appends its own error/result. Provenance now includes a *dynamic, remote-origin* record, not just static loci (Pottier's "static state vs dynamic stack", f7, but the dynamic half is now *on another machine*).
  - **dependency (surfacing):** the diagnostic is surfaced **not** on the probe line but on a *different* mutation line that **depends-on** the probe's result — a causal **link** across the dependency graph, not a containment/parent edge.
- f30. **A single surfaced annotation must co-resolve multiple distinct locators**, each answering a different user question: +SURE
  - `loc-host` — which host (+ probe session).
  - `loc-user-src` — the literal file path on the user's FS + line/span, **editable** (the thing they go fix).
  - `loc-probe` — the line in the **per-host mutated** probe-script (so they can debug *on the remote host*).
  - `loc-surface` — the dependent mutation line where it's shown (with its *own* `loc-user-src`).
  - `+` the parse-warn that **rode along** from an earlier stage (the `(result × [diagnostics])` accumulator of f1 is itself carried *through* the per-host transform and distribution, accreting more diagnostics at each stage).
- f31. **Prior-art gap → the bridge to gather next.** Domain-1 supplies the *per-tier* mechanisms (GCC: pack/chain one hop; Racket: transplant metadata across a transform; Roslyn: relative loci that survive edits + DAG-share; Wagner: diff-as-message + change bits + isolation; rustc: cheap-handle→lazy-resolve + confidence). +SURE **none** of it composes N tiers, forks per-host, hands off static→dynamic→distributed, or links to a surfacing site. That structure is the domain of: **W3C PROV-DM / data-lineage** (the derivation-DAG model: Entity/Activity, `wasDerivedFrom`/`used`/`wasGeneratedBy`); **OpenTelemetry** (trace = span *tree* across services; **span links** = causal-but-not-parent ≈ the depends-on surfacing edge; **baggage** + W3C Trace-Context = provenance propagated across the network); **composed/chained source-maps** (source-map-of-source-map; per-host fork = N maps from one input); **build-action provenance / remote execution** (Bazel REAPI action+worker+digest; SLSA attestations). This is the explicit domain-1→domain-3 seam, and it is higher-value than finishing tree-sitter/Lezer/SwiftSyntax depth.

Strawman provenance leaf (the OOB sidecar DESIGN.md concedes; grounds f29–f30):
```
diagnostic {
  payload:  warn|error|verdict
  derived:  [ # the transform lineage — FORKS per host at the probe tier
    { tier: user-src, file: "/home/u/site.dorc.sh", span: 12:4-12:31 },   # editable origin
    { tier: oracle,   ref: "nginx@sha256:…",         span: … },           # lifted-from
    { tier: probe,    host: web-7, span: 5:1-5:40, scriptHash: … } ],      # per-host mutated line
  ran_on:   { host: web-7, session: …, exit: 1, stderr: "…", at: … }       # dynamic, remote
  surfaced_at: depends_on -> { tier: user-src, file: …, span: 19:1 }       # the dependent mutation line
  rode_from: [ <parse-warn that survived the transform> ]                  # accumulated upstream
}
```

### bridge-B (front-1): distributed/forking provenance — PROV · OTel · SLSA (turn-4, 4 sources read)

- f32. **OTel independently arrived at di-2, and at Dorc's exact fork/join shape.** A trace is a DAG of spans with PARENT edges (single, containment) PLUS **LINK** edges (zero+, causally-related, may cross traces); the spec's **scatter/gather (fork/join)** guidance: when a root starts N downstream ops "all aggregated back in a single Span," that aggregating span **links to** the many it aggregates and "it is recommended *not* to set parent … as semantically the parent field represents a single parent scenario." +SURE the strongest external validation of di-2 (don't conflate containment with causal) *and* of the per-host-fork (scatter) + controller-fan-in (gather) structure — the prior-art independently reached "use a causal link, not a parent edge, for the aggregator." [B-otel-spec-overview-2024]
- f33. **W3C PROV gives the formal vocabulary for the whole N-tier forking DAG.** Entity (src-line / probe-image / runtime-result), Activity (probe-compile / probe-exec), Agent (controller / host / oracle); relations `used` / `wasGeneratedBy` / `wasDerivedFrom` / `wasAssociatedWith` / `wasInformedBy`; and **Plan** (a pre-defined procedure an Activity follows — the oracle/probe-template). +SURE Dorc's DAG *is* a PROV graph; di-2's edge-types are PROV relations (derived-from=`wasDerivedFrom`, ran-on=`wasAssociatedWith`+`used`/`wasGeneratedBy`, depends-on=`wasInformedBy`). PROV is domain-agnostic with extensibility points → a **Dorc PROV-profile** is the natural shape of the OOB record, and its agent/object/process triple-perspective covers loc-host / loc-source / the transform-chain. [B-prov-primer-2013]
- f34. **SLSA schematizes the runtime/distribution tier concretely — and its trust-split lands exactly on Dorc's oracle boundary.** A build = `builder.id` (host + Dorc) running a `buildType` template (the oracle/probe) over `externalParameters` (**untrusted**, MUST-verify) + `internalParameters` (**trusted** because the platform is) + `resolvedDependencies` (content-**digested** inputs the probe touched) → `subject`/`byproducts` + `metadata` (invocationId, started/finishedOn). +SURE three direct lifts: (a) the trusted/untrusted parameter split = the security-round oracle-grounding boundary (untrusted user-source verified; trusted Dorc-transform not) [→ `plans/102`]; (b) `ResourceDescriptor.digest` = prior-art for the parked content-hash version-drift gate [→ `../TODO.md`]; (c) "minimize externalParameters, maximize implicit-from-buildType" = keep the provenance leaf small by letting the oracle-template carry boilerplate. [B-slsa-provenance-2023]
- f35. **Calibration (the human's OTel caveat, first-party-confirmed): import the *idea*, not the machinery.** Cramer (Sentry founder, co-wrote a 2015 tracing spec): distributed tracing at its core is "structured events that carry two GUIDs — a trace-id and a parent-id … just building a tree"; a consumer "should only ever need span instrumentation, and in some cases forwarding the trace-id (baggage)"; everything else (logs, metrics, transports, collectors, semantic-conventions, the SDK) is design-by-committee scope-creep. +SURE Dorc needs *even less* than a tracing consumer — single-controller, no vendor-portability, no live-collector — so import only: span≈structured-event-node, parent-vs-link edges (f32), and a correlation-id threaded with the probe (≈baggage/trace-context); **do not** import OTel's collector/exporter/SDK/three-pillars. (The OTel spec even concedes tracing is "a cross-cutting concern [that] violate[s] … separation of concerns.") [B-cramer-otel-critique-2024] [B-otel-spec-overview-2024]
- f36. **Bridge-B closes the q-3 question: this is recording/linking a derivation graph, never distributed consensus.** All three model the *same* object — a content-addressed derivation DAG *built forward, queried backward* (di-3) — and none requires peer merge/voting: PROV asserts relations, SLSA attests one build, OTel's fan-in is a single aggregating span that *links* its sources. +SURE corroborates the turn-3 correction: Dorc's "merge" is a single-controller fan-in (OTel's gather-span), and the residual work is **correlation + backward-query indexing** (what PROV-AQ and OTel's trace-id grouping exist to serve).
- lead (ungraded, snippet-only — verify in bridge-A): the **lineage vs provenance** split — "lineage = flow (forward, what-flows-where), provenance = proof (backward, where-did-this-come-from)" — maps onto di-3's two directions *and* onto analyzer-graph (lineage/flow = the analyzer's dependency output) vs error-record (provenance/proof). --WONDER worth a graded source.

### bridge-B · depth (turn-5, 4 sources read) — PROV-AQ · Trace-Context · distributed-PROV · Bazel-REAPI

- f37. **PROV-AQ frames di-3 as keyed access + forward pingback.** Provenance is keyed by a stable **target-URI** (a stable id per DAG node); a **query service** returns provenance for a given target → backward resolution (runtime-failure-id → derivation chain → editable source). Separately, **pingback** lets an entity learn what was *later* derived from it → forward resolution (source → its downstream probes/dependents). > The parser enters a recovery state when receiving a token that is not expected in the current context. The recovery algorithm makes use of this property by not trying to backtrack in general: all the input that has been parsed is kept — even though the most satisfying correction for the user could have been to drop a token. The recovery will only try to complete the parse by filling holes, either by synthesizing values or by resuming the parse using tokens coming later.
