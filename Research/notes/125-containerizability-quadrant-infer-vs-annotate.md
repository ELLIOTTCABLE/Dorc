# notes/125 — Front B: the containerizability quadrant — infer vs annotate (first pass)

**Stamp:** round 12 · 2026-06-03 · Front B (`axis-quadrant`/`axis-infer`/`concl-quadrant`). The driving question (from `plans/121`): do shops *infer* the containerizable-vs-non-containerizable boundary, or *create+enforce* it artificially (tags/DSL/metadata)? f-numbers continue round 12 (f31+).
**Grounding:** [A-bazel-test-encyclopedia-2024], [B-saflate-network-assumption-inference-2022], [B-pytest-test-categories-network-isolation-2025], [B-kselftest-docs-2026], [B-k8s-e2e-best-practices-2023]; prior: notes/120 f1/f8 (size=resource-reach), `plans/077` (the seccomp network backstop).

## Findings

- f31. **The tier boundary is enforced by RUNTIME INTERCEPTION, not static inference — universally.** Bazel's sandbox default-denies network/fs; a test that is not `tags=["requires-network"]` and touches the net *fails* (`--sandbox_default_allow_network=false` to detect them) [A-bazel-test-encyclopedia-2024]. pytest-test-categories patches `socket.connect()` and, per the test's declared size, blocks (Small) / allows-localhost (Medium) / allows (Large) [B-pytest-test-categories-network-isolation-2025]. junit-airgap uses a JVMTI agent to fail on any socket/DNS. kselftest self-skips. Pattern: the tier is **hand-declared**, the violation is **auto-detected at the syscall/socket layer**. +SURE.
- f32. **That detection mechanism IS Dorc's `plans/077` seccomp backstop.** "Detect `socket(AF_INET)`" (plans/077, unprivileged seccomp, logs/notifies) is the *same* primitive as Bazel-sandbox / pytest-socket-patch / junit-JVMTI. So Dorc already banks, by design, the exact runtime-detection primitive the whole industry uses to police the network-reach boundary — and plans/077 already routes it to dual use (cost-class signal + undeclared-network backstop). For the network sub-axis, `axis-infer`'s "can Dorc detect it rather than annotate?" answer is **already YES**. +SURE.
- f33. **But the harder *containerizability* boundary (your f9) is HAND-ANNOTATED everywhere; nobody statically infers it.** K8s hand-labels environment-need via Ginkgo — `[Serial]`/`[Slow]`/`[Disruptive]`/`[Conformance]`/needs-real-cluster (migrating to `framework.WithSerial/WithSlow`) [B-k8s-e2e-best-practices-2023]; kselftest tests self-check "is this kernel feature/module present? else `$KSFT_SKIP`" [B-kselftest-docs-2026]. These encode "needs a real kernel/cluster/host a container cannot fake" — exactly the non-containerizable region — and they are all hand-declared or runtime-self-checked, **never inferred from source**. +SURE → your hypothesis holds for the containerizability axis: universally hand-annotated.
- f34. **The academic frontier INFERS the network dependency *dynamically* — explicitly because hand-annotation of it is unreliable.** saflate: "it is difficult or even impossible for engineers to write test fixtures or assumptions to reliably control or check the availability of networked resources"; so it instruments network-exception allocation sites + analyzes stack traces to infer the dependency and skip-with-provenance ("not black box… produces information as to why") [B-saflate-network-assumption-inference-2022]. Two transfers: (a) hand-annotation is the acknowledged weak point; (b) the working alternative is *observe-the-effect* inference, not static source analysis. +SURE.
- f35. **(concl-quadrant synthesis) Dorc is unusually well-placed on `axis-infer`, in two tiers — but full containerizability-inference is a research bet.** (i) **Network-reach sub-boundary:** Dorc can *detect* it at runtime via the `plans/077` seccomp backstop — banked, matching industry enforcement, and even feeding the optimizer's cost-class. (ii) **Full containerizability boundary** (the user's hard case — reconfigure sshd, needs-real-kernel): nobody infers it statically; Dorc's *only* lever beyond hand-`sh`-annotation is its **effect-analysis + oracle-effect-knowledge** — an oracle for `sshd`/`docker`/a-kernel-module could "know" which of its effects are non-containerizable, so containerizability would be inferred from *which oracle-effects a leaf's control-flow touches* (spelled-in-`sh` per `kOOB`, not a metadata tag). BUT that rests on oracles (parked), and even the academic frontier only manages *dynamic* inference of the *easy* (network) axis. ~SUSPECT. **Honest position:** network-reach tier is detectable/inferable today; the full containerizability tier is "annotate-via-sh-effects (K8s-label-style), with static inference as an oracle-dependent research bet — proven fallback is hand/sh-annotation."
- f36. **Teardown/restore is a first-class tier-discipline at the high-fidelity end (corroborates notes/120 f8).** K8s `DeferCleanup` (LIFO, restore-cluster-state) [B-k8s-e2e-best-practices-2023]; Bazel hermetic cleanup; Terraform reverse-order destroy; Test Kitchen create→converge→verify→destroy. Every real-environment tier pairs "spin up" with mandatory "restore" — a constraint Dorc's multi-host tier inherits (and one its plan/apply + idempotency model is already shaped for). +SURE.
- f37. **Size = a *mechanically-policed resource class*, reconfirmed from the canonical spec.** Bazel: a test's `size` sets both its timeout AND its assumed peak resource usage, and tests "SHOULD be hermetic… access only those resources on which they have a declared dependency," with runners reserving "the right to add such enforcement" [A-bazel-test-encyclopedia-2024]. This is notes/120 f1 (size=reach, policed) from the normative source — the precedent that a tier can be a contract the infra enforces, not a label the human is trusted to honor.
- f38. **(human corrections to f32/f35 — important; nothing decided)**
   1. **seccomp is two different things, and it is platform-limited.** `plans/077`'s seccomp is a *product* feature that detects network in the executed **user leaf** (sh/external program), which the human scoped as one *potentially-platform-limited* lint/DX tool **for oracle authors** — NOT an already-banked primitive for testing Dorc itself. Using seccomp to tier **Dorc's own (Rust) test suite** is a *different* application and is **Linux-only** (`axis-platform`; dev is macOS/Windows), so at most a **CI escape-detector**, never the hot-loop mechanism. For Dorc's own tests the portable primary enforcer is the **DST seam** — synthetic outcomes mean no real socket is opened, so "no network" holds by construction on every OS. f32's "already YES / already banked" overclaimed; corrected. (If seccomp is ever proposed as *core* self-test tooling it owes an explicit platform-limitation story + a concrete internal-use design — neither exists.)
   2. **Dorc is Rust, not sh — so "hand-sh-annotation" was the wrong frame for Dorc's own tests.** Disentangle two objects sharing the word "containerizable": **Object A** = a Dorc *self-test* (this round's unit-under-consideration) → **Rust**, tiered by **Rust** annotation (test attrs / cfg / nextest filters; the Bazel-tag/pytest-marker/K8s-label analog), infer-answer *mostly-no, like everyone*. **Object B** = a *user* leaf/oracle (sh) → its containerizability is a **product-feature** inference (effect/taint over user sh) where "spelled in sh"/`kOOB`/oracle-knowledge apply — **oracle-territory, parked**. f35's "infer from oracle-effects, spelled in sh" was Object B offered as if it answered Object A. They touch only at Dorc's *high-tier* integration tests, whose **sh fixtures'** containerizability (B) bounds whether that test containerizes; Dorc's low tiers (Rust unit + DST-seam sim) are pure Rust, Rust-annotated, portable.
   3. **Frontload for the conclusion (human):** lead with *the containerizability boundary is hand-annotated by everyone; nobody statically infers it* (the academic frontier infers only the network sub-axis, and only dynamically). The genuine-novelty bet is Object B (infer a user leaf's containerizability), oracle-dependent and parked; Object A (how Dorc tiers its own Rust tests) is conventional.
