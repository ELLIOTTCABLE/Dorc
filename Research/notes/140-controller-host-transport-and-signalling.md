## Findings (conversational; markers per CLAUDE.md)

### The SSH-native budget (tier-0) and the cost geometry
- f1. **Per *session channel*, SSH natively frames four lanes — free, no markers:** stdin, stdout (`CHANNEL_DATA`), stderr (`CHANNEL_EXTENDED_DATA`, `data_type=SSH_EXTENDED_DATA_STDERR=1` — separately framed, not bytes you demux), and exactly one integer exit-status (`CHANNEL_REQUEST` type `"exit-status"`, `uint32`; or `"exit-signal"`). RFC 4254 §6.10. The local `ssh` client re-raises the remote `$?`; it isn't inventing it. +SURE. **The SSH-native structured budget per command is exactly 2 byte-streams + 1 int.** Anything richer is the whole `kCOMMS` question.
- f2. **channel ≠ connection.** Many channels multiplex over one authenticated transport. New **channel** = one `CHANNEL_OPEN`/`CONFIRMATION` (~1 RTT over the already-encrypted/authed tunnel — no crypto, no re-auth, no new TCP). New **connection** = TCP + KEX (ephemeral-DH, asymmetric crypto, host-key verify) + auth — several RTTs of real crypto. +SURE.
- f3. **Two server ceilings** (already in `notes/072`): `MaxSessions` (default **10**) caps concurrent *session* channels per connection (exec/shell/subsystem; forwarding channels excluded); `MaxStartups` (default `10:30:100`) throttles concurrent *unauthenticated connections*, never channels. So 50 channels/conn → bounded by MaxSessions; 50 conns at once → throttled by MaxStartups + 50× KEX/auth. +SURE.
- f4. **ProxyJump asymmetry (load-bearing):** `ssh -J bastion target` is *stacked* SSH — KEX+auth to bastion, a `direct-tcpip` forward, then a *full second* KEX+auth to target inside it; m hops = m+1 nested handshakes. Once the nested tunnel exists, opening **another channel** is just messages over the built path (bytes traverse hops, but **no hop re-handshakes**) → multi-channel **hop-cheap**. Opening **another connection** re-runs the whole nested stack at every layer → multi-connection **hop-expensive** (what MaxStartups throttles). +SURE. *Corrects the "one connection vs N connections" mental model: N channels over one connection is a real, cheap third option.*
- f5. **No-pty required for the clean split.** `ssh -t` (any pty) wires the remote process's stdout+stderr both onto one tty *before* sshd sees them → merged, terminal control codes injected, stderr-split gone (exit-status still fine). Non-interactive `exec` (default) preserves the f1 split. +SURE. **Tension:** no-pty keeps the split but invites `isatty()`-triggered block-buffering (hurts livestream); pty restores liveness but destroys the split. The fd-split apply design therefore *commits Dorc to no-pty* + oracle buffering discipline (`stdbuf -oL`), not a tty. ~SUSPECT this is the constraint to bank.
- f6. **Native-SSH (russh / `x/crypto/ssh` / libssh2) makes multi-channel first-class** — one `Client`, many sessions, each read separately, no N CLI processes, no pty merge, no output-parsing. Does **not** lift the remote `MaxSessions` cap (target's policy). +SURE.

### The signalling ladder + the human's taxonomy correction
- f7. **Three tiers.** tier-0 = SSH-native (f1). tier-1 = in-band mux into the existing streams. tier-2 = a genuinely new channel the wrapped tooling is oblivious to. Dorc was leaning tier-1 by laziness; leaning away now.
- f8. **The correct dichotomy (human; my "stderr is a free OOB lane" was WRONG, retracted).** From Dorc's view *every* native lane — stdout, stderr, exit-status — already **belongs to the wrapped real program** (real tools make meaningful stdout/stderr/exit distinctions). So only two moves exist: **(a) multiplex** an occupied band — forces conflating Dorc's signal *with* the tool's content (classic in-band) or *compressing* the tool's own use to free a band; or **(b) establish a new channel** none of the tooling knows about. My stderr-sideband (`2>&1` the leaf, `printf >&2` Dorc markers) is **(a) a multiplex** — pays by collapsing the tool's stdout/stderr distinction — NOT OOB. +SURE (conceded).
- f9. **fd 3+ goes nowhere over SSH.** SSH transports only the remote command's fd 0/1/2; extra fds aren't carried. So there is no "just use another fd" — any fourth lane to the controller requires *establishing new transport* (2nd channel / forward / file-then-retrieve / 2nd connection). +SURE — the mechanism-level proof of f8.
- f10. **Establishing-new ≠ requires-an-executor; it splits by *liveness*.** *collected*-OOB: probe writes verdicts to a **private file** (pure POSIX `sh` — `mktemp` + `>>`), Dorc slurps over one cheap 2nd `exec` when the channel closes — truly OOB, executorless, single-connection, concurrency-safe (per-leaf files or `O_APPEND` short lines). *live*-OOB: signals arriving *as the probe runs* need an endpoint reading mid-run — a FIFO drained by a dedicated channel, a forwarded socket (needs remote `nc`/`socat` or a `/dev/tcp` bashism — a helper, not pure sh), or a bootstrapped executor. +SURE on collected; the live menu is the gather target.

### Phase asymmetry + knob couplings (with my own corrections)
- f11. **probe and apply have opposite concurrency profiles** → different carriers. apply: *(likely)* welded to **linear** execution of the elided CFG as ordinary `dash`, **livestreamed** (anti-Ansible: no buffer-and-dump). probe: massively **concurrent** (cross-host always; intra-host = open `kCONC`), and — corrected — **also wants live results**.
- f12. **CORRECTION (mine):** I claimed probe is "collected-not-streamed, so a private file is fine." The human wants to **build the plan in front of the user in real-time** (TUI: apply-lines greying as probe-results flow in, warning/error annotations appearing live; multi-host, possibly multi-channel-per-host). Literal command output may not show live, but **results + analysis-of-output must stream live**. So collected-OOB (f10) is too weak for the probe → re-opens **live-OOB**. This is *why* the charter exists.
- f13. **`kCONC` ↔ `kCOMMS` coupling.** stderr-multiplex under intra-host concurrency keeps per-leaf *verdicts* (short atomic `printf`s, leaf-id tagged) but loses *freeform attribution* (concurrent stdout shreds; no cross-stream ordering). An executor frames everything in one length-prefixed protocol, immune to that interleave — its *distinctive* value, ≈ its only one. So **wanting clean intra-host probe concurrency is the main thing that would justify an executor** (`kCOMMS-executor-OOB`); stay `kCONC`-linear and pure-`sh`+collected-file likely suffices. *(Human parked `kCONC` — overlaps `kFLATTEN`'s runtime half; left out of KNOBS for now.)*
- f14. **"4 things over 3 lanes" (justifies the human's channel-10 sketch).** To keep each probe channel a *pristine* mirror of real tool I/O (full rc + stdout + stderr — perfect off-ramp, zero Dorc bytes) **and** carry Dorc-status, you move 4 things over 3 native lanes (rc=exit-status, stdout, stderr = the tool's three; Dorc-status = the 4th, nowhere within the channel). So a 4th lane is genuinely necessary: a reserved status channel ("channel 10") fed by remote-side IPC, **or** accept a multiplex (f8a). +SURE the necessity; the mechanism is the charter.
- f15. **CORRECTION (mine):** "intra-host concurrency forces an executor" is false — collected concurrency is handled by per-leaf private files (f10). The executor is forced *only* by **live AND concurrent together** — precisely the probe's now-stated want (f12). Sharper reason than I first gave.

## Architecture options on the table (human's framing, refined past my stderr error)
- `arch-multiplex-inband` — pure-`sh`, one channel, Dorc markers conflated into stdout (or stderr-by-compression). Spoofable (f-sec), needs escaping. `kCOMMS-transpilation-inband`.
- `arch-collected-file` — pure-`sh`, private file + 2nd-channel slurp. Truly OOB, executorless, but **not live** → fails the f12 real-time-plan goal for the probe.
- `arch-live-oob-fifo` — pure-`sh` writes to a remote FIFO/file; a **reserved channel** (`cat`/`tail -f`) drains it live to the controller. *Candidate* executorless live-OOB — the charter's prime suspect. Multiplexes only Dorc-status onto the reserved lane; the N probe channels stay pristine.
- `arch-executor` — bootstrapped shim frames everything OOB over one channel (Ansible-python-style). Buys live+concurrent clean framing; costs the pure-`sh`/agentless property (`kAGENTLESS` push-welded but silent on a transient executor). `kCOMMS-executor-OOB`.
- f-sec. **In-band's recurring tax (→ `plans/102`):** data forges control. Blue-box 2600 Hz forged switch signals in the voice channel; GitHub Actions disabled `::set-output::` after untrusted stdout could forge workflow commands (retreat to `$GITHUB_OUTPUT` files). Any `arch-multiplex-inband` must nonce-delimit or accept the spoof surface; OOB lanes (file/FIFO/2nd-channel) are tooling-private by construction and dodge it. +SURE.

## Research charter (the gather — executorless live multi-probe OOB)
Best executorless mechanism + prior art for: **(1)** ultra-common primitives only (`ssh`, POSIX `sh`, coreutils — must work on stripped/BusyBox targets); **(2)** small-N parallel probes (say N≈9 channels, each carrying *pristine* per-probe rc+stdout+stderr) **plus** one reserved channel onto which all probe scripts emit Dorc status/metadata *live*; **(3)** keep the N read-clean-without-demux, multiplexing *only* the Dorc-controlled status onto the reserved channel. Systems crux: how does a script in channel-K (separate remote process) feed the reserved channel without polluting channel-K? Candidate mechanisms + prior-art to chase (the real tricks are likely obscure):
- **named pipe (`mkfifo`, POSIX) multi-writer → drained by a `cat` channel.** PIPE_BUF (≥512 POSIX / 4096 Linux) line-atomicity for clean multiplex of concurrent writers. (Stevens APUE; syslog-to-FIFO patterns.) Prime suspect.
- **the `--status-fd` convention** — `dpkg --status-fd`, `apt` (`APT::Status-Fd`), `gpg --status-fd`: machine status on a *reserved fd*, human output untouched. Domain-relevant (apt/dpkg!) semantic prior-art; the open part is transporting that fd over SSH.
- **debconf confmodule protocol** — line protocol between a config script and a controlling frontend over reserved fds; Debian-domain.
- **`bats` fd-3 convention** — reserve fd 3 for out-of-band display vs captured stdout.
- **tmux control mode (`-CC`)** — documented protocol multiplexing N pane streams + structured `%notifications` over one pipe.
- **Mitogen** (already in `sources.json`) — muxing many logical streams over one SSH channel + a tiny base64 first-stage bootstrap (executor-ish, but the mux + minimal-bootstrap technique is the art).
- **cdist / pdsh / ClusterShell / pssh** — how agentless parallel-SSH tools gather + attribute multi-node output.
- portable fd plumbing in POSIX `sh` (`exec 9>…`), `tail -f` vs FIFO tradeoffs (latency, persistence, open-blocking), whether a writable-FIFO is the reserved channel directly.
- **Counter-thesis to chase deliberately:** *why executorless live-OOB can't stay pure* — the cases (binary-safe payloads, backpressure, partial-write framing, reconnect/resumability, BusyBox `cat`/`tail` gaps) that force either an executor or a degraded guarantee.
