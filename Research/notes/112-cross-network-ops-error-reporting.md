## Findings

(f-numbers continue round 11's scheme; f1–f52 prior in `notes/110`.)

- f48. **The domain-3 thesis: the push/agentless tool most like Dorc (Ansible) has the WORST error-reporting, and it's mechanical.** Ansible's entire "error handling" page is CONTROL-FLOW (`ignore_errors` / `any_errors_fatal` / `max_fail_percentage` / block-rescue) with ZERO reporting / provenance / aggregation. The error taxonomy is *fragmented* (failed vs UNREACHABLE vs undefined-var vs syntax, each a separate mechanism — `ignore_errors` only catches a 'failed' result); "failure" is per-task *ad-hoc string/rc predicates* (`failed_when`); partial-execution + handler-skip silently leaves hosts inconsistent ("the configuration file might be changed but the service will not be restarted"). +SURE Ansible treats errors as control *decisions*, not *data* — the exact anti-pattern the `notes/110` model fixes, and the mechanical reason DESIGN.md says "Ansible is surprisingly bad." Dorc shares Ansible's push/agentless shape (`kAGENTLESS`), so this is the precise cautionary tale. [B-ansible-error-handling-2026]
- f49. **The structured-report tools INDEPENDENTLY converge on the `notes/110` model.** Puppet's `Transaction::Report` carries per-resource `resource_statuses`, **dependency-edge surfacing** ("Restarted: how many resources were restarted because their dependencies changed" = di-2's depends-on), **correlation UUIDs** (`transaction_uuid` ≈ Dorc run/correlation-id, `catalog_uuid` linking catalog→reports = bridge-B's `traceparent`, f38), **host** attribution, a **noop** flag (why-run), a *fail-safe* default ("@status = 'failed' until the report is finalized"), and a pluggable `Processor` (Json/Yaml/Rest — emission decoupled from the engine, cf. rustc Diagnostic-trait f17). +SURE the agent-based best-practice already reaches much of the model we derived — strong external validation, not invention. [B-puppet-transaction-report-2024]
- f50. **K8s gives the two-channel split + a three-valued, staleness-aware structured verdict.** Errors surface via **Events** (transient chronological log) vs **status.Conditions** (persistent structured status) — the two channels Dorc wants (a realtime diagnostic *stream* vs a durable *verdict*). `metav1.Condition` = {Type, Status ∈ **True/False/Unknown** (absence = Unknown — three-valued, matching Clang's ⊤ f23 and Dorc's best-effort "don't know"), **ObservedGeneration** (which spec-version the verdict observed → explicit STALENESS detection, cf. `kSTATE` / Merlin fs-invalidation f12), LastTransitionTime, **Reason** (programmatic id, required) + **Message** (human)}. +SURE the best structured-verdict model: three-valued, staleness-versioned, machine-id + human-message. [B-k8s-api-conventions-conditions-2024]
- f51. **Terraform grounds errors in the resource-DAG — the provenance backbone, in ops.** Nodes carry config/diff/state; edges are auto-derived from `depends_on` + interpolation-references; destroy/create split into separate nodes; cycles → mark-resources-failed + error (cf. Puppet). An error attributes to a node + its dependency edges = `loc-source` + the depends-on edge (di-2); the parallel DFS walk (a node is walked as soon as its deps are, semaphore-limited) = `kSCHEDULE`. +SURE Terraform's DAG = the Nature "backbone" (f39) realized in ops, the dependency-grounded attribution Ansible lacks. (The partial-apply / no-rollback state-ambiguity is the separate `kSTATE`/`kELISION` pain, not in this doc.) [B-terraform-graph-2024]
- f52. **(domain-3 synthesis) the opportunity is precise, and the model is validated.** The *structured* tools (Puppet report, K8s conditions, Terraform DAG) independently confirm the `notes/110` pieces — correlation-id, per-node structured *three-valued* status, dependency-edges, staleness-versioning, fail-safe, pluggable emission — while the *push/agentless* tool in Dorc's exact niche (Ansible) has none of it (errors = control-flow). +SURE So Dorc's domain-3 job is to bring the structured-report model to the push/agentless niche Ansible owns-but-fails — which is just *building the `notes/110` model*. Nothing here demands a new data-structure; domain-3 validates the one we have and names the bar: a Conditions-grade structured verdict **plus** a transient diagnostic stream. (Deferred: Chef handlers — pluggable post-run hooks with an ordering footgun, handlers miss pre-enablement failures; OTel-in-practice-for-ops.)

## Citations

> [B-ansible-error-handling-2026]:§(whole page) (relevance: +1:SURE) — all control-flow, no reporting; fragmented taxonomy; partial-state left unsurfaced
> When Ansible receives a non-zero return code from a command or a failure from a module, by default it stops executing on that host and continues on other hosts.
> The ignore_errors directive only works when the task can run and returns a value of 'failed'. It does not make Ansible ignore undefined variable errors, connection failures, execution issues (for example, missing packages), or syntax errors.
> If a task notifies a handler but another task fails later in the play, by default the handler does not run on that host, which may leave the host in an unexpected state. For example, a task could update a configuration file and notify a handler to restart some service. If a task later in the same play fails, the configuration file might be changed but the service will not be restarted.

> [B-puppet-transaction-report-2024]:class Puppet::Transaction::Report (relevance: +1:SURE) — per-resource status; dependency-restart; correlation UUIDs; host; noop; fail-safe default
> #resource_statuses ⇒ Hash{String => Puppet::Resource::Status} … A hash with a map from resource to status.
> Restarted: How many resources were restarted because their dependencies changed; Failed Restarts: How many resources could not be restarted
> #transaction_uuid … An agent generated transaction uuid, useful for connecting catalog and report. … #catalog_uuid … A master generated catalog uuid, useful for connecting a single catalog to multiple reports. … #host … The host name for which the report is generated. … #noop … Whether the puppet run was started in noop mode.
> @status = 'failed' # assume failed until the report is finalized

> [B-k8s-api-conventions-conditions-2024]:§Conditions (relevance: +1:SURE) — three-valued status; observedGeneration staleness; reason+message
> status of the condition, one of True, False, Unknown. … The absence of a condition should be interpreted the same as Unknown.
> observedGeneration represents the .metadata.generation that the condition was set based upon. For instance, if .metadata.generation is currently 12, but the .status.conditions[x].observedGeneration is 9, the condition is out of date with respect to the current state of the instance.
> reason contains a programmatic identifier indicating the reason for the condition's last transition … message is a human readable message indicating details about the transition.

> [B-terraform-graph-2024]:§"Building the Graph" + "Walking the Graph" (relevance: +1:SURE) — DAG nodes carry state; edges from depends_on + interpolation; cycle→fail; parallel walk
> Resource Node - Represents a single resource. … The configuration, diff, state, etc. of the resource under change is attached to this node.
> Explicit dependencies from the depends_on meta-parameter are used to create edges between resources. … References to resource attributes are turned into dependencies from the resource with the interpolation to the resource being referenced.
> Validate the graph has no cycles and has a single root. … To walk the graph, a standard depth-first traversal is done. Graph walking is done in parallel: a node is walked as soon as all of its dependencies are walked.
