# Phase 2 — probing deployment performance: fan-out ceilings, the controller bottleneck, thundering herd

The probe phase is *embarrassingly parallel by design* (note 70) — but "embarrassingly parallel" is an analysis property, not a deployment reality. The deployment reality is three hard ceilings (controller, network, target) and one self-inflicted hazard (thundering herd). Sources: Mitogen docs (A, primary), pyinfra docs (A, primary), OpenSSH (A), Salt docs (B), Twitter Murder (B), practitioner war-stories (B, graded individually; the `oneuptime.com/*` content-farm results were **rejected**).

## The single most important phase-2 finding: Dorc's compile-the-probe model sidesteps Ansible's worst bottleneck *for free* — but not all of it
**Ansible's per-task-SSH model** opens a connection (or multiplexed channel) and round-trips **per task per host**: N tasks × H hosts = N·H round-trips. Mitogen's whole reason to exist is killing that overhead — and the numbers show the overhead, *not the work*, dominates fleet ops:
- Mitogen (primary doc): **"Mitogen cannot improve a module once it is executing; it can only ensure the module executes as quickly as possible."** The win is *all* orchestration overhead. "**A single network roundtrip** is used to execute a step whose code already exists in RAM… saves **4 ms runtime per 1 ms of network latency** for every playbook step." Process reuse saves **300–800 ms per step**. Testimonials (real fleets): **45 min→3 min, 1.5 hr→3 min, 19m49→7m36, 14 min→2 min.** "**Most potent on playbooks that execute many short-lived actions, where Ansible's overhead dominates the cost of the operation.**"
- **Dorc's probe is exactly "many short-lived actions"** (read-only checks) — the precise regime where orchestration overhead dominates. But Dorc avoids it *structurally*, not with a connection hack: the analyzer **pre-bundles all checks into one probe script per host**, shipped once, run locally, streaming results back = **one round-trip per host**, independent of check-count. This is the *static* analog of Salt's persistent-minion / Mitogen's persistent-interpreter — achieved **without an agent**, because the work-shaping moved to compile time. The compile-the-probe architecture *is* the performance story for phase 2.
- **What it does NOT fix:** the controller still opens **one connection per host** (H connections), so for large H the controller-side and target-side ceilings below still bind. Compile-the-probe turns N·H → H; it doesn't turn H → 1.

## Ceiling 1 — the controller (the agentless-push tax; this is the architecture fork)
Agentless-push-over-SSH (the #1 Ansible-loved sacred cow — *keep it*) **inherently makes the controller the bottleneck.** The trap, stated by practitioners as a *lock-in*: Ansible's **fork-per-host + per-task-SSH was an early design choice, baked into the execution model — "hard to retrofit… the reason Mitogen exists as a connection-layer replacement rather than a config tweak."** Concretely:
- **Fork-per-host**: each parallel host = a forked process (a Python interp in Ansible's case) + a file descriptor + an SSH connection. **Memory grows linearly with parallelism.** Default `forks=5` (absurdly low); practitioners raise to 50–100+ and provision a beefy controller. Single-controller practical limit: **hundreds to low-thousands of hosts.**
- **The fix is async I/O, not more forks**: `parallel-ssh` (libssh2 non-blocking, many connections in one thread) pushes the controller ceiling from hundreds to **thousands** per controller. **→ Dorc design redline: the executor must be async/event-driven (non-blocking many-connection I/O), NEVER fork-per-host.** This is the single most important controller-side choice and it's an *architectural* one (retrofit-hostile, per the Ansible lesson) — decide it at the executor seam now, even though the executor is SEAM-not-core (orchestration-go-no-go). **Concrete ceilings to respect** (parallel-ssh war-stories, B): default `ulimit -n` is often **1024** file descriptors (each connection ≥1 fd); a `select()`-based event loop **hard-caps at 1024** fds — so use `epoll`/`kqueue`, not `select`, or you inherit the 1024 wall *in the async path too*; raise `ulimit -n` as part of controller setup. Quantified Ansible controller cost (Red Hat capacity-planning, B): **~1 CPU core per 50–100 simultaneous connections**, plus per-fork RAM.
- **Separate the analysis plane from the execution plane (Ansible AAP + Puppet compile-masters both did this, *late*).** At scale, the CPU-heavy *compile/analyze* step contends with the I/O-heavy *push* step on one controller. Puppet's catalog compilation became the master bottleneck → "compile masters"; Ansible AAP "separated control and execution planes… scale each independently." → **For Dorc the analysis (compile-the-probe) and the executor (push/stream) are *already* distinct components (orchestration-go-no-go); keep them separable as a deployment concern from day 1** so a heavy analysis run doesn't starve the live push, and so each can scale/host independently. Retrofit-hostile if fused.
- **Beyond one controller — the fan-out tree, AND its documented lock-in.** Salt's **Syndic** (master-of-masters → syndics → minions) and Mitogen's **Connection Delegation** (push via bastion, "code executed on every intermediary, data cached and re-served") tier the push to scale horizontally. **But the fan-out tree is NOT free — Salt's syndic is a cautionary tale** (SaltStack GitHub issues #55564 "syndic architecture does not scale", #43003, #19864, all grade B but primary/maintainer): the syndic **"sends results sequentially, one by one, waiting for reception-confirmation of each — ~100 messages/minute over a high-latency link"**, so over-WAN syndics deliver events "**with hour-long delays… that part of the infrastructure is rendered useless until the bottleneck clears**," and the **maintainers stated they won't fix it** (it's an enterprise/PR-only concern). → **The "wish I'd known" for Dorc's fan-out hook: the bottleneck migrates from connection-fan-out to *result-aggregation* — a relay that re-fans the probe must also *gather and forward results*, and naive sequential per-result confirmation re-serializes exactly what the tree parallelized.** Design the relay's result path as **batched/streamed/async aggregation**, not per-result ack. The agentless analog (a per-subnet relay host that copy-and-runs the self-contained probe, controller→k→(N/k)) is *easier to fan out* than Salt (the probe is stateless) but inherits the *same result-aggregation hazard*. Reserve the hook; v1 flat (low-thousands suffices for homelab/dotfiles); the user's "entire org over SSH" worst case needs it, designed with the syndic lesson baked in.

## Ceiling 2 — the network + the target's sshd (the limits nobody mentions until they bite)
- **SSH connection setup is expensive**: key exchange (DH/Curve25519) is ~1–2 RTTs ≈ **50 ms on LAN, 300–500 ms via bastion/VPN at 100 ms RTT**. Across thousands of connections this is real wall-clock. **ControlMaster/ControlPersist** multiplexing reuses the TCP+SSH connection (avoids re-handshake) — **30–60% reduction for multi-task runs** (OpenSSH cookbook, A). Dorc's one-round-trip-per-host already minimizes this, but ControlMaster still helps the probe→apply→re-probe sequence to the same host (reuse the connection across phases).
- **The target sshd has hard ceilings** (Microsoft/OpenSSH docs, A): **`MaxStartups` default `10:30:60`** — only **10 concurrent *unauthenticated*** connections, then probabilistic drop (30% at 11+), hard reject at 60. **`MaxSessions` default 10** — max multiplexed sessions per connection. → **Footgun: a controller fanning out faster than targets authenticate gets connections *dropped*, not queued** — silent partial failure that looks like flaky hosts. This bounds *connection rate*, not just count. Mitigations: pace new-connection rate per target; multiplex (one connection, many sessions, up to MaxSessions); back off on drop. *This is a "wish I'd known" — the limit is on the **target** you don't control, and the default is low.*
- **pyinfra's blunt truth** (primary doc): "**The major bottleneck is almost always the network connecting the local machine to the target hosts.**" Default 20 parallel; "connecting to a large number of hosts from a single machine can hit local system limits around open files and network connections." pyinfra also validates Dorc's two-phase shape: it runs a **"fact gathering" phase then an "operation execution" phase** — "this two-phase approach means pyinfra knows what it will do before it does it." (Dorc = probe-phase then apply-phase; the model is field-proven.)

## Ceiling 3 — shared backend resources: the thundering herd (self-inflicted DDoS)
The probe is read-only and cheap, but **probes/mutations that consult a shared resource turn embarrassingly-parallel into embarrassingly-parallel-load**:
- **The user's expensive-remote-controller example = Mitogen's documented `local_action` serialization footgun**: delegating many targets to *one* machine/account causes **artificial serialization — slowdown = `task_duration × num_targets`** (Mitogen primary doc). → **An oracle/probe that calls out to a central or third-party service (not purely local to the target) serializes the whole fleet through that service.** Design rider (ties to note 74 cost model): oracles must be **local-to-target by default**; any oracle that consults a remote/shared service is a *serialization + load hazard* and must carry a cost-class so the optimizer keeps it under a guard (don't fan it out unnecessarily) and the executor can throttle it.
- **Package mirror / registry herd**: H hosts pulling from one apt mirror / container registry / license server simultaneously = self-DDoS. Mitigations (practitioner consensus, B): **`throttle`** (cap concurrent hosts for the resource-touching step), **stagger/jitter** start times, **local caching proxy** (apt-cacher-ng / Squid / Pulp), **pre-stage** artifacts. Ansible's `throttle` keyword exists *specifically* to protect shared rate-limited resources.
- **Artifact distribution becomes THE bottleneck at scale, independent of the orchestrator** (Twitter **Murder**, B): naive "every host pulls from origin" is **O(N)** and saturates the origin; **BitTorrent/peer-to-peer** spreads distribution across the fleet's own bandwidth, **O(N)→~O(log N)** — Twitter reported **"dozens of minutes → less than a dozen seconds" (~75× faster)**; Apache-licensed, with a maintained single-command successor (`russss/Herd`). → **Dorc cedes artifact-fetching** (apt's job, orchestration-go-no-go), but the **derived advantage**: Dorc's fact-domain partition (`pkg:foo` etc., note 53) means **Dorc knows the herd size — which hosts need which artifact — *before* applying.** So it can *advise/throttle/point-at-a-cache* without building distribution. The cost model needs a **shared-resource dimension** (which probes/mutations contend on the same backend), not just per-host cost. Keep probe + oracle bundles **small and content-addressed** so shipping *them* never becomes its own herd.
- **Splay/jitter is the canonical herd cure across the whole field** (Puppet/Chef pull model, B): the default 30-min run-interval × thousands of agents hammered the master, so **`splay` (a random delay within a window)** spreads connections over time. Four independent tools (Ansible `throttle`, Salt, Puppet/Chef `splay`, k8s rate-limits) converge on **jitter + cap** as the herd defense. → Dorc's probe scheduler should **jitter probe start-times across hosts by default** when a step touches a shared resource (derived from the fact-domain), not fan out simultaneously. Cheap, universally-blessed, prevents the self-DDoS.

## The resumability dividend (Dorc gets both rivals' best property, neither's cost)
Partial failure mid-rollout is the **normal case at scale, not exceptional**. The recovery models:
- **Ansible**: re-run from top; idempotence no-ops converged hosts — but **re-pays the full check cost** (no state); band-aids are `--limit @retry_file` and `start-at-task`.
- **Terraform**: **state** records what succeeded → resumes without re-work — but pays the **central-state tax** (contention, secrets-in-state, the liability we reject).
- **Database-backed idempotency stores** (the distsys-pattern answer, B) hit the **"phantom write" problem**: a 3-state key (`IN_PROGRESS`/`COMPLETED`/`FAILED`) that crashes *between* doing the work and writing `COMPLETED` faces "an impossible decision on retry — did the original complete or not?" → **Dorc sidesteps this entirely**: it doesn't trust a stored key, it **re-probes reality**. "Did the mutation complete?" is answered by *observing the host*, not by a durable flag that can be stale/torn. This is a third, subtle advantage of probe-over-state: **no idempotency-key/state-write atomicity problem, because the source of truth is the host, recomputed.**
- **Dorc = both rivals' best, neither's cost**: re-probe is the recovery (no central state, like Ansible) **but cheap and parallel** (unlike Ansible's sequential re-check) because the probe is compiled + embarrassingly parallel, and the analysis already knows which sections are independent. **The probe IS the retry-file — *derived*, not stored.** After partial failure, re-probing finds exactly the still-dirty set fast. Maps onto the two modes (AGENTS contract matrix): `reconcile` re-probes all; `update` re-probes the declared scope. *A genuine differentiator that falls out of the architecture for free.*

## Monitoring/pull-collection systems = the probe phase in another domain (Prometheus / osquery)
Prometheus (pull, scrape-on-interval) and osquery/Fleet (distributed SQL over thousands of endpoints) are **exactly Dorc's probe phase**: read-only fleet-state collection on a schedule, collector-as-bottleneck. Four transferable findings (sources: Prometheus docs A, Robust Perception A, osquery docs A, Grafana/Last9 B):
- **The pull-model "up" advantage — and a fleet-level elision-soundness point.** Pull *knows the inventory* (via service discovery), so a target that fails to respond is **known-down**, not silently absent; push can't distinguish "healthy but quiet" from "dead." → **Dorc's probe is pull, so it inherits this advantage — and must use it: an unreachable host is `unknown` (⊤ / can't-skip), NEVER counted as "clean/skipped."** A fleet-scale elision-soundness rule I hadn't stated: *non-response ≠ converged.* The inventory (which hosts *should* exist) is therefore load-bearing for soundness, not just for addressing — it's how you know a missing host is missing. (Confirms inventory-as-SEAM, orchestration-go-no-go, with a soundness reason.)
- **Staleness semantics — the constraint that bounds cross-host memoization (note 75).** Prometheus injects an **explicit staleness marker** (NaN) when a scrape fails / a target vanishes, and uses a **5-minute lookback** — deliberately refusing to show last-known-value as current, because *stale-shown-as-fresh is dangerous for alerting.* → **This directly constrains the cross-host probe-verdict memoization lever (note 75): a reused/cached verdict MUST carry a freshness bound (TTL / staleness marker); skipping on a stale memoized "clean" verdict is an elision-soundness violation** (the host may have drifted since). The two levers *compose*: build-systems memoization gives cross-host reuse; monitoring-staleness bounds its validity window. **A memoized verdict is `(verdict, content-key, freshness)` — reuse only within the freshness bound, else re-probe.** This is the missing safety rail on the memoization win.
- **Target-side resource safety (osquery watchdog) — the ceiling I under-covered.** osquery runs a **watchdog that kills queries exceeding CPU/mem thresholds**, because "you can't have monitoring degrade the thing being monitored." → **Dorc's probe runs on production hosts; an expensive probe must be resource-bounded *on the target*, not just rate-limited on the controller.** The probe needs a **target-side watchdog / resource cap** (and a ⊤-on-watchdog-kill: a probe killed for cost is `unknown`, not `clean`). New design rider; the target is a constrained resource too, and the fail-direction must stay safe even when the *target* gives up.
- **Differential reporting (osquery differential mode).** Scheduled queries report only **changed rows since last run**, not the full result — slashing data volume. → **Dorc re-probe can report only the *delta* from the last probe** (changed facts), shrinking fan-in (compose with backpressure below + memoization above). The probe's natural output is "what changed," not "everything" — which is *also* exactly the plan/diff UX.
- **Sharding the collector** (hash-based or functional, Prometheus) is the **fan-out-tree finding confirmed in a third independent domain** (after Salt syndic + Mitogen delegation) — strong convergence that horizontal collector-sharding is *the* answer past single-collector limits.

## The observability↔scale tension (the realtime-output hard requirement has a fan-in cost)
The realtime-streamed-output *hard requirement* (AGENTS) makes the controller a **fan-in aggregator**: thousands of hosts (fast producers) → one controller (slow consumer). This is the dual of the fan-*out* problem and has its own well-mapped failure mode (distsys log-aggregation literature, B):
- **Backpressure is the core hazard**: without it, the controller's buffers grow unbounded as fast producers outrun the slow consumer → memory exhaustion → cascading failure. **The debuggability sacred cow is not free at scale.**
- **The cures mirror the fan-out cures** (pleasing symmetry, reuse the machinery): **bounded buffers + drop/sample** (shed low-priority output under load — but *never* drop the verdict/error, only verbose progress); **decouple ingest from processing** (a durable queue so a host never blocks on the controller); **hierarchical aggregation** (the **same fan-out-tree relays aggregate results on the way back** — directly the Salt-syndic result-path lesson above, and the reason to design that relay's aggregation as batched/streamed); **per-stream rate-limit** at the source.
- → **Dorc design rider:** the streamed-output path needs **backpressure + a priority split (verdicts/errors always; verbose progress droppable)** from the start; the fan-out-tree relay is *also* the fan-in aggregator (one tier, both directions). The probe's compactness helps (one structured result per host, not a live byte-stream per task) — prefer **structured, bounded per-host result messages** over raw streamed stdout where possible, reserving raw streaming for the interactive/debug (`update`/single-host) case where fan-in is small. *This is the concrete content of synthesis decide-now item #8.*

## Footgun list (phase-2 "wish I'd known")
1. **Fork-per-host executor** → linear-memory controller wall at hundreds of hosts; retrofit-hostile. **Redline: async/non-blocking I/O from the start.**
2. **Flat single-controller push** → caps at low-thousands; the user's worst-case ("whole org over SSH") needs a **fan-out-tree hook** reserved now (probe is self-contained → relaying is trivial for Dorc).
3. **Fanning out faster than `MaxStartups`** → silent connection *drops* on the target you don't control (default 10:30:60). Pace connection rate; multiplex within `MaxSessions` (default 10); back off on drop.
4. **Oracles that consult a central/remote service** → fleet serializes through it (`task_duration × num_targets`). Keep oracles local-to-target; cost-flag the remote ones.
5. **Ignoring the shared-backend herd** → self-DDoS the apt mirror/registry/license server. Derive contention (fact-domain partition), throttle/jitter, point at caches; don't build distribution but don't be blind to it.
6. **Large/uncached probe or oracle bundles** → shipping them becomes its own herd. Keep them small + content-addressed.
7. **Treating ControlMaster as optional** → re-handshake cost (50 ms LAN / 300–500 ms bastion) × thousands. Reuse connections across probe→apply→re-probe.
8. **Unbounded fan-in of live output** → controller buffer exhaustion (the realtime-output requirement's hidden scale cost). Backpressure + priority-split (verdicts/errors never dropped, verbose progress droppable) + structured-bounded-per-host results over raw stdout streaming except in the small-fan-in interactive case.
