## Findings

(f-numbers continue round 11's scheme; f1–f47 in `notes/110`, f48–f52 in `notes/112`.)

- f53. **The dominant error is in the INPUTS (cardinality), not the cost FORMULA — keep the cost-model simple, invest in input quality.** Leis disentangled the optimizer empirically: cardinality estimators "routinely produce large errors" (worse with more joins), and "the influence of cost model errors is dwarfed by cardinality estimation errors — even quite simple cost models seem to be sufficient." +SURE directly tunes `Q-COSTVEC`/`kPRECISION`: don't over-engineer the cost FORMULA; the dominant error is in the input estimates (how-expensive-is-this-op / how-much-does-this-elision-save), so simple-formula + good-inputs beats fancy-formula + guessed-inputs. [A-leis-query-optimizers-2015]
- f54. **Don't over-rely on estimates; prefer ROBUST strategies that don't blow up when wrong.** Leis: "query performance is unsatisfactory if the query engine relies too heavily on these estimates"; hash-join + full-scan engines are "quite robust even in the presence of large cardinality estimation errors," while more plan choices (indexes) make misestimates more damaging. +SURE the core lesson for a best-effort analyzer whose estimates *will* be unreliable (Dorc makes the same wrong independence/uniformity assumptions on correlated ops-data): prefer strategies that degrade gracefully when wrong over aggressive ones that win-big-if-right / lose-big-if-wrong. [A-leis-query-optimizers-2015]
- f55. **Robustness = graceful degradation, no cliffs — and the strongest move is to REMOVE the brittle decision.** Haritsa: robustness ≈ "graceful degradation as opposed to performance cliffs"; robust *operators* — SmoothScan morphs index↔seq-scan from observed selectivity *without* cardinality estimation; G-join unifies the join algorithms so "the optimizer doesn't have to choose, preventing mistakes." +SURE two Dorc imports: (a) "no cliffs" is literally Dorc's no-cliff / gradual-enhancement thesis; (b) where the analyzer can't reliably decide (probe-vs-just-run, `kPROBING`), prefer an *adaptive* strategy that can't be catastrophically wrong over a brittle static choice. [B-haritsa-robust-qp-2020]
- f56. **PlanBouquet = the theoretical anchor for "probe to discover, don't statically estimate" — with provable worst-case bounds.** Haritsa: PlanBouquet "completely ABANDONS the brittle selectivity estimation process, replaced with a calibrated 'trial-and-error' DISCOVERY mechanism," giving provable Maximum-Sub-Optimality guarantees (SpillBound: MSO ≤ D²+3D, D = #error-prone predicates). +SURE **this is Dorc's probe phase, formalized**: rather than statically estimate host state (and be wrong), *discover* it by a bounded calibrated probe — and the prior-art shows discovery-over-estimation can yield *provable* worst-case bounds. The strongest external justification for the probe-first architecture (`kPROBING`, `kFAIL-withhold`). [B-haritsa-robust-qp-2020]
- f57. **The communication mechanism: EXPLAIN = "show the plan + the cost-estimates," and the estimated-vs-actual gap is the deopt signal.** Postgres EXPLAIN renders the plan tree with each node's `(cost, rows, width)` estimates (the top-line total is what the planner minimizes); EXPLAIN ANALYZE adds actuals, and the rows value "is the only value shown in both estimated and actual — that lets you quickly find erroneous cardinality estimates." The cost "only reflects things the planner cares about … it cannot change [other costs] by altering the plan," and "every correct plan outputs the same row set, we trust." +SURE Dorc's plan/apply preview, made precise: show planned probes/applies + Dorc's cost-estimates; after running, surface the estimate-vs-actual gap as the deopt diagnostic; the cost-model weighs ONLY what Dorc's decisions can change (correctness invariant — `kFAIL` — only performance traded). [B-postgres-explain-2024]
- f58. **The suboptimal-warning channel EXISTS but is widely IGNORED because imprecise — the cautionary tale.** SQL Server plans embed typed WARNING nodes (SpillToTempDb, ColumnsWithNoStatistics, PlanAffectingConvert, NoJoinPredicate, UnmatchedIndexes, MemoryGrantWarning…), split compile-time(predictive) vs runtime-only(observed). But Darling (SQL Server MVP) shows most are false-positive-prone — "NoJoinPredicate" fires even when a predicate exists ("the check is only at the Nested Loops Join … no attempt to track pushed predicates further"); convert-warnings fire when irrelevant; "everything up to the actual version feels futile and useless, and seems to lie to you." +SURE **design-affecting**: a "you're leaving perf on the table / write an oracle" channel is only as valuable as its PRECISION; over-conservative warnings (the natural output of incomplete/local static analysis = Dorc's exact regime) train users to ignore ALL of them (warning-fatigue / cascade-trust, cf. f10 + rustc warning-fatigue guidance). Dorc's deopt-hints must be precise or silent. [B-darling-plan-warnings-2022]
- f59. **(domain-2 synthesis) the RDBMS planner is the closest mature analogue to Dorc's analyzer/probe-optimizer, and it says: estimate humbly, discover by probing, communicate the gap, warn only precisely.** +SURE four pillars: (1) simple cost formula + good inputs, don't over-rely on estimates (f53/f54); (2) prefer robust/adaptive strategies, no cliffs (f55); (3) where estimation is brittle, *discover by calibrated probing* with bounded suboptimality — Dorc's probe phase, justified (f56); (4) communicate via an EXPLAIN-style plan+estimates+actuals, and gate suboptimal-warnings on precision or they're noise (f57/f58). Methodology caveat (Leis): validate Dorc's cost-model on REAL correlated ops-data, not synthetic that shares its independence/uniformity assumptions (cross-link the corpus work `notes/080`–`086`). ### domain-2 · how EXPLAIN is *implemented* — the round-relevant deep-dive (turn-9; PostgreSQL instrumentation source)

The human's refocus: for *this* round (continuously-gathered + tracked metadata) the relevant thing about EXPLAIN is its *implementation*, not the optimization theory. Read the PostgreSQL executor instrumentation source (`instrument.h` + `instrument.c`).

- f60. **EXPLAIN ANALYZE is the round's exact pattern: a SEPARATE per-node accumulator struct threaded alongside execution, joined to the static plan node only at print time.** The static `Plan` (estimates) and `NodeInstrumentation` (actuals — `total`/`startup`/`ntuples`/`nloops`/`nfiltered1-2`/buffers/WAL) are *distinct* structures; EXPLAIN walks both trees in lockstep to emit estimate-vs-actual. +SURE this is f1 (`result × accumulated-diagnostics`) and f14 (cheap static node + side-accumulator keyed by node-identity) realized in a mature engine — don't fold runtime metadata into the plan node, keep a parallel accumulator. [B-postgres-instrument-2026]
- f61. **No per-leaf instrumentation-overhead problem exists for Dorc — provenance is controller-side reconstruction, not per-leaf runtime instrumentation. (Corrected twice, turn-9/10 — supersedes BOTH the "this is kFIDELITY" over-claim AND the "per-leaf is a network round-trip" error.)** PG's opt-in (`INSTRUMENT_*` bitmask + `ExecProcNode` wrapper-swap, "to avoid overhead in the normal case") is perf-motivated by a clock-read *per tuple* in a CPU-bound Volcano loop (millions of pulls; +SURE mechanism, ~SUSPECT the ~2× factor). **Dorc's leaf is neither that nor a network op:** the compiled-probe (`notes/072`) batches all a host's leaves into ONE tunneled script + output-stream, so network is **O(hosts)** (~2/host, ~100 total), and the leaf runs as `sh` on the host *inside the streamed batch*. The real per-leaf cost is a *third regime* — output-stream bytes + controller parse — and rich provenance never rides the stream: the controller compiled the probe, holds the probe↔source map (bridge-A f47, q-1), and attributes delimited output chunks itself; the host emits dumb delimited output. → there is **no costly per-leaf runtime instrumentation to gate**; "carry a minimal in-stream marker, resolve richly at the controller" (= f14, rustc `Span`→`SourceMap`) is the cheap default, and the only lever is marker *compactness* (the q-1 stream-balloon concern). (A small real cost reappears only for per-leaf *timing* actuals — a host-side clock-read per leaf into the stream — a candidate for faithful-mode opt-in, but tiny vs PG's per-tuple version.) The only structural transfer from f60 stands: keep the accumulator *separate* from the static node — for Dorc motivated by cleanliness/immutability, not perf. [B-postgres-instrument-2026]
- f62. **Multiplicity is carried provenance — but PG's *resolution* (per-loop AVERAGE) is a cautionary CONTRAST for Dorc, not a pattern to copy. (Refined per human pushback, turn-9.)** PG: a node reruns (`nloops` = nested-loop rescans), so the displayed actual rows = `ntuples/nloops` (per-execution average), making it comparable to the per-execution estimate — a raw total would falsely read as an N× misestimate. +SURE the *principle* transfers: a measurement from a leaf that ran more than once is meaningless without "how many times / across what." BUT PG averages because its rescans are **homogeneous** (same scan, same data); Dorc's signature multiplicity (per-host fan-out) is **heterogeneous** (hosts differ — that's why you probe each; some are slow / time out), so averaging would destroy the signal (one failed host hidden behind eleven fast ones). → For heterogeneous fan-out, keep the **per-host breakdown + count, never pre-average**; only homogeneous multiplicities (single-host retry loop, uniform `for`) may be averaged PG-style. Rule: multiplicity is carried, never silently summed; the allowed reduction depends on homogeneity (tag each dimension homogeneous-or-not, average only the homogeneous ones). [B-postgres-instrument-2026]
- f63. **Aggregation up the tree is a first-class named primitive (`InstrAggNode`) — and it's how parallel workers' stats fan in.** `WorkerNodeInstrumentation` holds a per-worker array of `NodeInstrumentation`; `InstrAggNode(dst, add)` merges one into another to roll worker stats up to the leader. +SURE bridge-B's controller fan-in (f36) at single-node scale: the merge of per-executor records into one is a *named, reusable* operation, not a bespoke step — Dorc's per-host→controller roll-up is the same `InstrAggNode` shape. [B-postgres-instrument-2026]
- f64. **The accumulator tracks DERIVED, reason-tagged diagnostics, not just raw counts: `nfiltered1`/`nfiltered2` = tuples removed by scanqual/joinqual vs "other" quals.** The instrumentation records *why* tuples disappeared (which predicate class filtered them); EXPLAIN surfaces "Rows Removed by Filter." +SURE the runtime record should carry not just "what happened" but "what was filtered/skipped and the reason-class" — for Dorc the per-leaf record should note what the probe *elided/skipped and why* (elision provenance), exactly as the planner records what each qual removed. [B-postgres-instrument-2026]
- f65. **(round-relevant takeaway, corrected turn-9) EXPLAIN's value to THIS round is its instrumentation ARCHITECTURE — read net of two over-claims I retracted.** +SURE the solid transfers: a side-channel accumulator kept *separate* from the static node (f60 — for Dorc: cleanliness/immutability, *not* perf); a named aggregate-up primitive (f63 ≈ bridge-B fan-in); derived reason-tagged diagnostics (f64 ≈ the elision/why record — the cleanest, cost-free transfer). The two corrections: PG's opt-in instrumentation is perf-motivated by a per-tuple CPU loop that **inverts** for network-bound Dorc — so it is *not* `kFIDELITY`, and Dorc should just always-instrument (f61); and PG's per-loop *averaging* is a cautionary contrast — Dorc's heterogeneous per-host fan-out must keep the breakdown, not average (f62). Net: the query *executor* corroborates the separate-accumulator + aggregate-up + reason-tagged-diagnostic shape, but its perf-driven optimizations are a **disanalogy** (PG's CPU-per-tuple hot loop vs Dorc's O(hosts) batched-probe with controller-side provenance — a leaf is not a network op), *not* a validation of Dorc's perf knobs. ## Citations

> [A-leis-query-optimizers-2015]:Abstract + §1 + §1-end (relevance: +1:SURE) — cardinality dominates; cost-model dwarfed; rely-too-heavily is bad; robustness of simple engines
> We investigate the quality of industrial-strength cardinality estimators and find that all estimators routinely produce large errors. We further show that while estimates are essential for finding a good join order, query performance is unsatisfactory if the query engine relies too heavily on these estimates.
> the influence of cost model errors is dwarfed by cardinality estimation errors and that even quite simple cost models seem to be sufficient.
> Query engines that mainly rely on hash joins and full table scans, are quite robust even in the presence of large cardinality estimation errors. The more indexes are available, the harder the problem becomes for the query optimizer resulting in runtimes that are far away from the optimal query plan.

> [B-haritsa-robust-qp-2020]:§1 + §2.1 + §2.3 (relevance: +1:SURE) — bad plans are the norm; remove the decision (SmoothScan/G-join); PlanBouquet abandons estimation for discovery
> [Lohman 2014] The wonder isn't "Why did the optimizer pick a bad plan?" Rather, the wonder is "Why would the optimizer ever pick a decent plan?"!
> SmoothScan [7] is an adaptive access path that … continually morphs between the sequential scan and index access options … [G-join] brings the popular join algorithms (nested-loops, sort-merge and hash) into a common framework. This merger makes it unnecessary for the optimizer to have to choose between the alternatives, thereby preventing mistakes.
> In this execution technique, called PlanBouquet, the brittle selectivity estimation process is completely abandoned, and replaced with a calibrated "trial-and-error" discovery mechanism. This technique lends itself to provable MSO guarantees even in situations where state-of-the-art systems may suffer from arbitrarily poor execution.

> [B-postgres-explain-2024]:§14.1.1 (relevance: +1:SURE) — plan tree + cost estimates; cost reflects only what the planner can change; correctness invariant
> The output of EXPLAIN has one line for each node in the plan tree, showing the basic node type plus the cost estimates that the planner made … The very first line … has the estimated total execution cost for the plan; it is this number that the planner seeks to minimize.
> the cost does not consider the time spent to convert output values to text form or to transmit them to the client … the planner ignores those costs because it cannot change them by altering the plan. (Every correct plan will output the same row set, we trust.)

> [B-darling-plan-warnings-2022]:§(whole) (relevance: +1:SURE) — warning channel exists but false-positive-prone → ignored
