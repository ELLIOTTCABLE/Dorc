# IFDS/IDE engine + the side-effect & Datalog bridges (Q1 mechanism, Q2/Q3 connections)

## What IFDS is, and why it's the engine candidate
SURE: IFDS (Reps–Horwitz–Sagiv'95) solves **precise interprocedural dataflow in polynomial time** by reducing it to **reachability along interprocedurally-realizable paths** (paths whose call/return symbols form a balanced-parenthesis/Dyck string). Restrictions: facts `D` finite; transfer functions **distribute over meet** (union/intersect). Covers gen/kill (separable) *and* non-separable problems (truly-live, copy-constant-prop, possibly-uninitialized).
- **Exploded supergraph**: each CFG node × `(D ∪ {0})`; each transfer function is a small graph (≤`(D+1)²` edges); **function composition = relational composition**, so the whole analysis = graph reachability.
- **Tabulation algorithm** (worklist): accumulates **path edges** (suffixes of realizable paths from `main`-start) and, crucially, **summary edges** — "how the dataflow value *after* a call depends on the value *before*." **Summary edges ARE compositional procedure summaries, computed on the fly.** This is the same compositional-summary idea Facebook scales (note 51), here with a precise algorithm.
- Cost: `O(E·D³)` worst case; for **h-sparse** problems (each statement touches a small slice of state — the *normal* case) it collapses toward `O(E·D)`. "each statement changes only a small portion of the execution state" — true for shell commands too.
- **Precision (1B)**: realizable-path (matched call/return) ⇒ "meet-over-all-*valid*-paths," strictly more precise than naive all-paths, *with no soundness loss*. And IFDS naturally supports **demand queries** ("does fact `d` hold at point `n`?") — minimal work per query (Q1B + Q3).
- **IDE** generalizes IFDS: facts are *environments* (symbol→value maps) with distributive environment transformers — needed for value-carrying problems (constant-propagation-like). Same tabulation shape.

## The two bridges that make IFDS load-bearing for Dorc
1. **Side-effect/MOD analysis is expressible in/near IFDS.** The paper's §7 shows **Callahan's flow-sensitive side-effect problems — `must-modify` and `may-use` — are IFDS(-adjacent)**: `may-use` is a locally-separable IFDS problem; `must-modify` is a same-level-valid-path problem; both solved by the Tabulation Algorithm in `O(E·D)`. → **Dorc's "may-mutate" is literally a `may-modify` side-effect problem**, which this framework handles precisely and cheaply. The reachability-∩-may-mutate probe set (Q1) is a `may-use`/`may-modify`-style IFDS query.
2. **IFDS ≡ a Datalog/deductive-database computation.** Reps showed dataflow-as-deductive-DB; the paper states the Tabulation Algorithm "has a straightforward implementation as a logic program." → This is the bridge to the **queryable-fact-base** model (Souffle/Doop/CodeQL, note 53): we can express the analysis as Datalog rules over program-fact relations, get the IFDS-style summaries as derived relations, and get **Q2 (retain/query facts, add analyses as new rules) for free**, with Souffle/IncA giving **Q3 scale + incrementality**.

## How this composes with the rest (the forming spine)
- **Engine = IFDS/IDE-style fact propagation** (reachability on realizable paths) **for the "is this mutation reachable, and what facts does it establish/kill"** layer — distributive, finite-fact, demand-able, summary-based.
- **Domain = a side-effect/MOD lattice** (Salcianu–Rinard shape, note 51): the *characterization* of *which* state a command touches (`pkg:foo`, `file:/etc/x`, `svc:foo#enabled`). NB: the points-to-style *characterization* is **not** itself distributive/finite in general, so the engine is likely a **hybrid**: IFDS/IDE for the reachable-fact layer + a separate (coarser, finite) effect-class/fact domain for the "what is touched" layer. Flag this as a real design decision for synthesis.
- **Soundness bias (note 51)**: over-approximate — unknown context / non-distributive blowups → ⊤ (un-probeable + can't-skip). IFDS's meet-over-valid-paths is exact *within* the finite-distributive fragment; everything outside collapses to ⊤. This is exactly the "bounded model, absolute over-approximation inside it" stance.
- **Scale (Q3)**: summary edges = per-role/per-module summaries; compositional ⇒ parallel + incremental; an ops corpus is embarrassingly modular ⇒ ideal fit.

## Caveats / open questions for synthesis
- Is Dorc's full probe-extraction expressible as ONE IFDS/IDE problem, or a *pipeline* (reachability via IFDS; effect-characterization via a points-to-like pass; relevance via slicing)? GUESS: a pipeline, sharing one CFG/supergraph + fact base.
- The "realizable path"/Dyck machinery matters once we have **functions/`source`** (interprocedural). For purely intra-function skip (Tier A, the 90%), plain intraprocedural monotone dataflow (SPA §5) suffices — IFDS earns its keep at the cross-function/Tier-B tier.
- IFDS requires the meet to be union/intersection and functions distributive; the planning-log's "establish/require/conflict facts" gen/kill *is* distributive, so the dependency-fact layer fits cleanly; the version/precision refinements may push toward IDE.
