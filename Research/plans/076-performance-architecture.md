## 0. One-paragraph thesis
Dorc's performance story is **"move the work-shaping to compile time, then schedule the unavoidable remainder using the structure the analysis already computed."** The analyzer is bounded and well-understood (cubic-per-unit, made fleet-scalable by compositional summaries + diff-time incrementality, kept off the EXPTIME context-sensitivity cliff and under the memory wall by recency abstraction). The probe phase wins *structurally* — compiling all checks into one shipped-and-run-locally probe turns Ansible's N·H round-trips into H, the precise regime where orchestration overhead dominates — but still faces three hard ceilings (controller, network, target-sshd) that demand an **async, fan-out-capable executor decided now** (retrofit-hostile). The mutation phase is a **derived-DAG scheduling problem** (list-scheduling / critical-path / resource-constrained), where Dorc's edge is that it *computes* the inputs (DAG width, shared-resource contention, the still-dirty set) that Ansible/Terraform make you hand-author — so it can auto-tune what they hand-tune. The connecting discipline is the perf reframe (note 70): **every shortcut is elision-safe (costs probes/runs, never a wrong skip), so Dorc can be tuned aggressively toward speed in the safe direction** — the one thing perf can *never* trade is the per-phase fail-direction (probe: withhold; apply: perform). The single highest-value *new* lever the round surfaced (note 75, build-systems lens): **cross-host probe-verdict memoization** — the fleet is a cloud-build cache, so probe the equivalence-class and reuse verdicts across identical hosts (fleet-linear → equivalence-class-linear), bounded by a Prometheus-style freshness rail.

## 1. The reframe that makes perf safe to chase (from note 70)
Performance is a **cost-and-latency problem for Dorc, not a correctness one** — because the two soundnesses (AGENTS §1) are preserved by *orientation*, not by precision. Cutting corners for speed costs precision = more probes / more re-runs = slower, never wrong, **provided every shortcut fails toward the conservative end of whichever soundness governs the phase** (probe: don't-execute; apply: do-execute). This *inverts* the usual perf-vs-soundness triangle (which binds under-approximating bug-finders): Dorc may aggressively time-box, sample, widen, and cap, where a bug-finder cannot. **The cost asymmetry is the whole license to optimize.**

## 2. Phase 1 — analyzer runtime (detail in note 71)
**Where the cliffs are, and that Dorc sits below them:**
- **Cubic is the floor** (Heintze–McAllester, 2NPDA-complete) for CFL-reachability/points-to/may-mutate — *accept it*; the analyzed unit is a small, sparse role/function, so cubic-in-(small-n) is cheap. IFDS is **O(ED³)** but collapses toward **O(ED)** on the h-sparse normal case (shell commands touch few facts).
- **Context-sensitivity is EXPTIME — but the k-CFA paradox says Dorc's data model probably dodges it.** Van Horn–Mairson: k-CFA (k≥1) is EXPTIME-complete. *But* Might–Smaragdakis–Van Horn (PLDI 2010): the **same spec is polynomial for OO / flat-data and exponential only for functional/closure data** — the blowup needs environments that *recombine variables across contexts* (closures). **Dorc's abstract state is a flat fact-map, not closures → plausibly the polynomial-amenable regime, so per-host/per-role context-sensitivity may be *affordable* (a usable precision dial), not fatal.** Redline restated precisely: default context-insensitive; **add per-host/per-role context freely *as long as no construct lets one unit's context capture-and-recombine another's fact-environment*** (that closure pattern is what detonates). Verify the flat-domain assumption in the spike. (Note 71 detail.)
- **Memory is the first real wall** (TAJS OOM at 512 MB; Doop/Soufflé materialized-fact RAM). **Recency abstraction is a memory lever as much as precision** (non-optional on two axes). The Datalog-vs-IFDS-vs-hybrid substrate decision is partly a *memory-ceiling* choice — measure peak RSS in the spike, not just runtime.
- **Scale is structural and proven** (Facebook: 100M LOC < 30 min/24 cores, ~1T LOC/day, compositional ⇒ naturally incremental, 0%→70% fix-rate at diff-time). The ops corpus (R roles × H hosts) is *easier* than that. **Dorc's scale risk is not corpus size; it's accidentally choosing a non-compositional or context-sensitive design.**

## 3. Phase 2 — probing deployment (detail in note 72)
**The structural win:** compile all checks into **one probe per host, shipped once, run locally, streaming back** = one round-trip per host regardless of check-count — the static, agentless analog of Salt's persistent-minion / Mitogen's persistent-interpreter, in exactly the "many short-lived actions" regime where Mitogen shows orchestration overhead dominates (45min→3min-class wins). It turns **N·H → H**; it does not turn **H → 1**, so:
**The three ceilings (and the decisions they force):**
- **Controller (the agentless-push tax):** fork-per-host = linear-memory wall at hundreds of hosts, and it's the **canonical retrofit-hostile lock-in** (Ansible's own lesson — Mitogen exists *because* you can't config-tweak it away). **Redline: async/event-driven non-blocking I/O, never fork-per-host.** For the "whole org over SSH" worst case, **reserve a fan-out-tree hook** (delegated/tiered SSH relays; trivial for Dorc since the probe is self-contained).
- **Network + target sshd:** SSH handshake is 50 ms LAN / 300–500 ms bastion → use **ControlMaster** across probe→apply→re-probe. **`MaxStartups 10:30:60` and `MaxSessions 10` are hard target-side ceilings you don't control** — fanning out faster than targets authenticate gets connections *silently dropped*; pace connection rate, multiplex within MaxSessions, back off on drop.
- **Shared backend (thundering herd):** an oracle that consults a central/remote service **serializes the fleet through it** (`task_duration × num_targets` — the user's expensive-controller example = Mitogen's `local_action` footgun). Package-mirror/registry herds are self-DDoS; at scale, **artifact distribution itself becomes the bottleneck** (Twitter Murder: O(N)→O(log N) via P2P). Dorc **cedes** distribution but **derives** the herd (fact-domain partition knows who-needs-what *before* applying) → advise/throttle/point-at-cache. **Cost is a vector, not scalar:** (local-cost, reach-frequency, shared-resource-id).
**The resumability dividend:** re-probe is the recovery — **Ansible's no-central-state + Terraform's know-what's-done, with neither's cost** (cheap because compiled + parallel; the probe *is* the retry-file, derived not stored). Falls out for free; a genuine differentiator.
**Monitoring/pull-collection lens (Prometheus/osquery = the probe phase in another domain, note 72):** (a) **pull knows the inventory → an unreachable host is `unknown` (must-probe / can't-skip), never silently "clean"** — a fleet-level elision-soundness rule (*non-response ≠ converged*); inventory is load-bearing for *soundness*, not just addressing. (b) **Staleness bounds the memoization lever**: a reused/cached verdict MUST carry a freshness/TTL (Prometheus refuses to show stale-as-fresh); a memoized verdict is `(verdict, content-key, freshness)`, reuse only within the window — the safety rail on §4b's cross-host memoization. (c) **Target-side watchdog**: the probe runs on production hosts; bound its CPU/mem *on the target* (osquery's "don't degrade the monitored thing"); a watchdog-killed probe is `unknown`, not `clean`. (d) **Differential re-probe**: report only changed facts since last probe (shrinks fan-in; *is* the plan/diff UX). (e) collector-sharding confirmed in a third domain.

## 4. Phase 3 — mutation orchestration (detail in note 73)
**Dorc derives the DAG the others make you author** → achievable apply-parallelism = **DAG width**, which the analysis *computes* (Terraform's "parallelism=50 won't manufacture concurrency the graph doesn't have" — but Dorc derives the graph). **Precision unlocks concurrency**, tying phase-1 precision to phase-3 throughput.
- **Use the scheduling theory, don't reinvent it:** list-scheduling (Graham (2−1/m)-approx), **critical-path-first** prioritization (not source-order), **RCPSP** for shared-resource constraints (NP-hard → heuristics, not optimality). **Graham anomaly (surprising):** more workers / relaxed constraints can *increase* makespan — so **the schedule matters more than the raw parallelism width**; don't expose width as the only knob.
- **The batching/blast-radius/partial-failure knobs are mature scars** (Ansible `serial`/`max_fail_percentage`/`throttle`/`run_once`/`linear`-vs-`free`; k8s `maxSurge`/`maxUnavailable`/readiness-gate). They are **executor SEAM**, but Dorc **derives their inputs** (batch boundaries from the dependence graph, throttle groups from fact-domain contention, canary set from blast-radius) → **auto-tune what they hand-tune.** The k8s **readiness-gate = re-probe between batches** (mutate → re-probe-converged → advance).
- **The under-investment trap (the user's central fear), named:** pouring effort into "avoid work" and shipping a naive parallel-for executor that loses to Ansible the instant real ordering/resource/failure constraints appear. **Mitigation: the executor seam carries the rich knobs in its *design* from day 1** (even if v1 implements a subset); the value proposition is *not* "we avoid work" but "we schedule the unavoidable work better because we computed its structure."

## 4b. Build systems are the skip-thesis prior art (detail in note 75)
Dorc's MH1 *is* a build system's job (skip work that needn't redo, derived from a dep-graph + change-detection). "Build Systems à la Carte" (Mokhov–Mitchell–Peyton Jones) [A-mokhov-build-systems-a-la-carte-icfp-2018] gives a vocabulary + leverage:
- **Adopt the Scheduler × Rebuilder factoring.** **Scheduler** (topological/restarting/suspending) = Dorc's apply-phase DAG walk (§4). **Rebuilder** (dirty-bit / verifying-trace / constructive-trace / deep-constructive) = Dorc's skip-decision. *Name them this way.* The formal **minimality** (run at most once, only if deps changed) + **correctness** properties = Dorc's elision goal + two soundnesses, restated.
- **NEW PERF LEVER — cross-host probe-verdict memoization (constructive traces over the fleet).** Verifying-trace = local (per-host prior record); **constructive-trace = keyed by content-hash of inputs → reuse a verdict computed *anywhere***. The fleet is the cloud-build setting: probe the **equivalence class** (hosts sharing the relevant fact-slice content-hash), not all H hosts → **fleet-linear probing becomes equivalence-class-linear.** The key is *derived from the dependence analysis* (it already knows the fact-slice a verdict depends on). **This rescues the "Dorc loses on homogeneous fleets" boundary below** (probe a handful, reuse across the identical rest). SUSPECT high-value; design the rebuilder as **constructive/cross-host**, not just verifying/per-host. Decision: memoization-key precision (reuse-rate vs soundness — an under-keyed verdict reused on a secretly-different host = elision-soundness violation).
- **Content-addressing seats the two modes.** `update`/git-diff = **input-addressed** ("source changed ⇒ re-examine"; over-skips cosmetic-equivalent, under-skips cosmetic-different); the **probe = content/state-addressed early-cutoff** over it. → confirms (with build-theory) the earlier conclusion: **the probe, never the git-diff, is the skip authority.** Modes are the input- and content-addressed *layers* of one system, not rivals.
- **Early cutoff = an un-named apply-time precision lever.** Mutation *ran* but the resulting **state-fact is unchanged** (idempotent re-write → identical file) ⇒ dependents keyed on that fact **still skip.** Recovers skips a naive "it ran ⇒ downstream re-runs" forfeits; composes with change-signal-as-value (handlers dissolve into this).
- **Hermeticity = non-determinism exclusion, independently validated.** Bazel/Nix caches are correct *only if hermetic* (output a pure function of declared inputs); non-determinism breaks the cache exactly as it would break Dorc's skip. **Settled principle 3 is not a Dorc shortcut — it's a necessary precondition for *any* sound skip system.** Borrow the framing: **"hermetic oracles"** (a verdict is a pure function of canonicalized host-state).
- **"Build without the bytes"** → compute verdicts **host-local**, return only the verdict (not raw gathered facts that are only a local intermediate) — minimizes fan-in (§3 observability tension).

## 5. Two boundary findings (the honest "when does this pay off")
- **Probe-vs-just-run (refines MH1):** for a *cheap* idempotent mutation (`mkdir -p`, a one-line file write), the probe (stat + decision + round-trip) can cost **more** than just running it idempotently. So Dorc is **not** "always probe then skip" — it's "**probe only when probing is cheaper than the work it might save / safer than running blind.**" Cheap-idempotent → just run; expensive (`apt install`) or dangerous or remote (`curl`) → probe first. This is a cost-model decision at the *opposite* end from batch-vs-guard, and it should be first-class. (SUSPECT this meaningfully shrinks the probe and is a perf win, not just a nicety.)
- **When Dorc loses (the boundary condition), and how memoization narrows it:** the approach pays off when **(work is expensive OR dangerous) AND (skip-rate is meaningful).** On a fleet of identical, fast-converging hosts with cheap operations, the probe *was* **pure overhead** — but **cross-host memoization (§4b) rescues much of this case**: probe the equivalence class, reuse across the identical rest, so even a homogeneous fleet pays ~one probe, not H. What remains genuinely Dorc-losing: a *small* homogeneous fleet of *cheap* ops with *no* reuse structure, where `pdsh 'script.sh'` already wins. The unanalyzable-rate — the fraction of leaves that collapse to must-probe (note 71 / phase-1 Step −1) — and now the *fleet-homogeneity / reuse rate* set the ceiling. **Honest positioning: Dorc is for expensive/dangerous/heterogeneous fleets, OR large homogeneous fleets (where memoization wins) — not small homogeneous cheap-op fleets.** Measure skip-rate, cheap-mutation fraction, AND fleet-equivalence-class structure on the real corpus.

9. **Rebuilder shape: constructive/cross-host, not just verifying/per-host** — the cross-host memoization lever (§4b) is an *architectural* property (the verdict store + content-key must exist from the start); bolting cross-host reuse onto a per-host design later is a rewrite. Design the verdict as content-addressable + the key as dependence-derived now, **and carry a freshness/TTL on every verdict** (the Prometheus-staleness rail; a verdict is `(verdict, content-key, freshness)` — without the freshness field designed in, memoization is unsound and retrofitting it touches every cache read). `[corpus-sensitive: fleet homogeneity / equivalence-class size = reuse rate; the corpus may say "don't bother"]`

### Your call — gated by fleet-size ambition + how much orchestrator you cede; the corpus is silent
1. **Async, non-blocking executor** (never fork-per-host; `epoll`/`kqueue` not `select` — the 1024-fd wall lives in the async path too; raise `ulimit -n`). — *Ansible's literal lock-in.*
5. **Fan-out-tree hook reserved** in the executor seam (flat single-controller caps at low-thousands; the worst-case is org-scale) — **designed with Salt's syndic lesson baked in: the relay's *result-aggregation* path must be batched/streamed/async, never per-result sequential-ack** (that re-serializes what the tree parallelized; Salt's syndic does this and "does not scale", maintainers won't fix). Easier for Dorc than Salt (the probe is stateless to relay), but the aggregation hazard is identical.
6. **Executor rich-knobs as a first-class designed seam** (rolling/throttle/blast-radius/readiness/restartable), not an afterthought parallel-for. — *the under-investment trap.*
8. **The observability↔scale tension** (now researched — distsys fan-in/backpressure literature): the realtime-output *hard requirement* makes the controller a fan-in aggregator (thousands of fast producers → one slow consumer); unbounded, its buffers exhaust → cascading failure. **Decide now:** backpressure + a **priority split (verdicts/errors never dropped; verbose progress droppable)** + **structured bounded per-host result messages** as the default (reserve raw stdout streaming for the small-fan-in interactive/`update`/single-host case) + **the fan-out-tree relay doubles as the fan-in aggregator** (one tier, both directions; batched/streamed, per the syndic lesson). The debuggability sacred cow has a real, designable scale cost.

### Just-do-it — dominant-strategy or near-free; not really a choice
2. **Compositional + incremental analysis from commit 1** (not whole-program batch). — *Facebook's lesson; retrofit = rewrite.*
1b. **Analysis-plane / execution-plane separable** from day 1 (heavy compile-the-probe must not starve the live push; scale/host independently). — *Ansible AAP + Puppet compile-masters both retrofitted this late.*
4. **Versioned on-disk summary/IR format** (cached summaries must survive across runs for incrementality *and* be shareable across a team/org). — *the analyzer's own diff-time + oracle/role sharing both need it.* **Caveat (rust-analyzer's lesson): version the format now (cheap), but treat cache *persistence* as an optimization-with-cold-start-fallback, not a v1 given** — rust-analyzer, the incremental poster child, *punts* on-disk persistence (too subtle to invalidate correctly); Dorc's diff-primary, smaller corpus makes cold-start tolerable for v1. Incremental recompute itself is **~80%-cheap, measured** (Reviser).
10. **"Hermetic oracle" contract** — the oracle-purity contract (AGENTS engineer-audience) should explicitly include *determinism/canonicalization* (verdict = pure function of declared, canonicalized host-state), because it's the precondition for *both* soundness *and* cross-host memoization. State it in the oracle contract from day 1; retrofitting determinism onto a corpus of non-hermetic community oracles is impossible.

## 7b. Effect-observation altitude (calibration + derivation)
If Dorc observes *what a command actually did* (calibration-harness state-deltas; future Dozer-style oracle/effect derivation; drift-witnessing), do it at **eBPF altitude (<1–2% overhead), never ptrace (2–10×, up to 102× on syscall-heavy loads)**, and keep it **offline/fixture-side** (the calibration container), never on the live probe path. On-probe self-profiling (timing/exit-codes) feeds the cost model; syscall-level observation is a *calibration/derivation* tool. Offline eBPF tracing on fixtures can also *suggest* an oracle's effect-class/cost-class — feeding the cost model's tier-2 hint **without engineer annotation** (note 74).
