# Orchestration feature go/no-go (Space 2 — integrate vs delineate)

> ⟢ 2026-06 — the "deliberately AVOID a central state backend" call is now the knob `kSTATE`, **reopened** (Terraform-central-state vs rust-analyzer-stateless-recompute are the two poles; not a settled "never"). The `update`/`reconcile` modes are `kELISION`. Grep the slug for current handling.

Criterion (user's): **INTEGRATE** iff the feature benefits from conflation with {analysis/extraction engine + sh-superset language/parser + core evaluation-planner} enough to keep in mind while planning. **DELINEATE** iff cleanly separable AND (limited benefit to our target uses OR well-covered by do-one-thing-well external tools). Three resolved buckets below (INTEGRATE / SEAM = delineate-but-reserve-the-interface-now / CEDE). Each call is grounded in the user-study (notes 61–62), the planning log, and the round-2 analysis architecture. Stance: Dorc is **not an orchestrator** (planning log) — but eyes-open about the eventual orchestrator goal, so a few get a reserved seam rather than a flat cede.

## INTEGRATE — these ARE the value, or are inseparable from the engine/language/planner
- **Idempotency / change-detection** — the thesis itself; *derived* by the analysis engine (round 2), not a bolted-on feature. (Ansible-loved; the sacred cow we must keep.)
- **Plan / check / diff mode** — the derived check-projection *is* `plan`; round-2 provenance *is* the plan's "what + why." Terraform's plan/apply is the validated UX target. Core.
- **Drift detection** — *redefined as the same probe*: "host diverged from intended end-state" = the read-only skip-check. Don't build a separate drift subsystem; it falls out. (Directly answers the Ansible user-study's deletion/drift gap — I3: "deletes are hard… nothing knows what to undo.")
- **Facts / host introspection** — *redefined as the demand-driven probe* (round 2): gather only the state the analysis asks about, on demand. Do **not** build Ansible-style upfront `gather_facts` (a named perf anti-pattern in the user-study). Facts = targeted probes.
- **Handlers / notify / change-reactivity** — dissolves into native control flow (`if changed; then reload; fi`); the change-signal is a first-class analysis value (planning log). Not built — falls out of language+engine.
- **Error handling / blocks / rescue / control flow** — native shell (`set -e`/`trap`/`||`); the CFG analysis must model these anyway (round-2 hazards). Native, not a feature.
- **Retries / until / delay (convergence-by-retry)** — first-class language construct (planning-log Bucket-D): not derivable from a bare script, reality is flaky, and the retry condition *is* a check (ties to the contract). Maps to the user-study's rollback/recovery wish.
- **Variables** — native shell variables (deletes Ansible's 22-rule var-precedence + Jinja, the #1 usability pain). *(Config-file templating is split out — see CEDE.)*
- **Selective execution (tags / `--limit`)** — *obsoleted, not built*: per-function/per-host skip is **derived** (round 2); tags/`--limit` are the hand-maintained version Dorc computes. Keep only a thin user-facing "just this role/host" selector in the planner. That selector *is* the two-modes distinction (AGENTS "contract matrix"): `dorc update` (goal-framed — correctness for the declared scope, e.g. the git-diff; speed elsewhere; the hot-loop) vs `dorc reconcile` (state-framed — reconcile all, elide only probe-proven; the final push). Modes change elision *scope*, never elision *soundness*.
- **Realtime streaming output / structured logging / good error messages** — planning-log hard requirement AND the user-study's #2 theme (debugging opacity: "stuck 15 min with no output," "I transfer complex stages to a shell script to debug"). Tied to the executor + round-2 provenance. A core differentiator, not a nicety.
- **Module/provider knowledge** — *this is the oracle layer*: the engine is generic over per-command knowledge (round 2 / CoLiS). The "do the actual work" knowledge is the pluggable Rules + community-authored oracles — the core, not ceded.
- **Oracle↔tool-version binding (MH2)** — analysis-coupled (which `.check` applies to `foo@3.5`); integrate the version-binding logic. *(The registry mechanism itself is delineated — see CEDE.)*

## SEAM — delineate the component, but design the interface now (the executor/transport tier + eventual-orchestrator hooks)
- **Transport / connection (ssh + ControlMaster multiplexing)** — a **thin built-in ssh-streaming executor** (planning log), separate component over the IR/verdict seam, swappable for pyinfra/etc. The analysis doesn't need it; the realtime-output requirement does. Keep agentless-ssh (the #1 Ansible-loved trait, n=9). Reserve the seam; keep it do-one-thing-well.
- **Inventory / host & group management** — DELINEATE the inventory *system*; **read** existing sources (ssh config, existing inventories, a flat host file). The per-host fact-instantiation (round 2) needs only a host *list*. Low benefit to conflate; well-covered. (Inventory pain was 30% of the user-study's perf theme — don't reinvent it, consume it.)
- **Fleet execution policy (serial / rolling / throttle / `run_once` / delegation)** — DELINEATE to the executor, but **reserve, don't preclude** (planning log): `serial` rolling-batch is a real production need; the parallelism *comes from* compositional summaries (round 2), rolling is just batching the apply. Executor hook, not analysis.
- **Async / poll / long-running** — DELINEATE (executor concern: fire-and-check-later for a 40-min `apt dist-upgrade`). Interacts with realtime-output but the async mechanism is executor-side. Reserve a hook; low analysis benefit.
- **Apply-the-plan artifact** — INTEGRATE the *transient* plan artifact (Terraform's "a plan it can later execute, aborting if assumptions changed"), but explicitly **NOT** a persistent state store (see CEDE: state backend).
- **Collection/oracle distribution** — SEAM: the community/distribution is the existential risk (planning log), so design the *sharing format* now, but **reuse existing registries** (git, OCI, or piggyback npm/apt per MH2) rather than building a package manager.

## CEDE — clean separation; low benefit and/or covered by do-one-thing-well tools (or deliberately not-built)
- **Become / privilege escalation (sudo/doas/runas)** — transport/identity; well-covered. The analysis only needs to *know* a command needs privilege — that's an oracle annotation / a Tier-B `permission` fact (round 2), not a Dorc subsystem.
- **Secrets / vault** — CEDE to do-one-thing-well (sops, age, Vault, 1Password, ansible-vault-compat). Strong external ecosystem; conflating secret-management with an analysis engine is scope creep + a security-surface mistake. Seam: consume already-resolved values/refs. (Matches the user's self-hosted do-one-thing-well preference.)
- **State backend / locking (Terraform-style)** — **deliberately AVOID.** Round-2 + user-study verdict: Terraform's monolithic state is its *biggest liability* (contention, 15–30 min plans, drift, secrets-in-state). Dorc's whole point is to **probe real host state** instead of maintaining a central state file — there is no shared state to lock or contend. This is a positive design choice, not a gap.
- **Provider/resource graph + declarative dependency ordering** — CEDE the declarative model. Dorc *derives* ordering from imperative control-flow + fact-dependencies (round 2 / planning log). No declarative resource graph to author; the dependence graph is computed. (Anti-declarative stance — keep it.)
- **GUI / control plane / scheduling / RBAC (AWX/Tower, Terraform Cloud)** — CEDE hard. AWX is the user-study's **#1 "lack of support" pain (20.9%)** and releases paused since 2024; a control-plane/scheduler/RBAC/UI is a huge separate product, well-covered by CI + cron + existing schedulers. Integrate only a clean CLI + machine-readable output so external CI/schedulers drive Dorc (the round-2/Facebook **diff-time deployment** model). Do-one-thing-well: be the analyzer+planner+executor a CI job calls, not the platform.
- **Config-file templating engine (Jinja-equivalent)** — CEDE to unix tools (`envsubst`/`m4`/`jq`/`gomplate`); Dorc provides only the *convergence-predicate + plan-time diff* around them (planning log: file-editing is the cleanest defer). Notably **deletes** the user-study's #1 usability pain (Jinja/YAML escaping hell).

## The shape that falls out
**INTEGRATED core** = {analysis/extraction engine + sh-superset language/parser + evaluation-planner + oracle library + realtime/provenance output}. **SEAM** = a thin, swappable ssh-streaming executor (+ reserved hooks for rolling/async/distribution) reading existing inventory. **CEDED** = secrets, privilege-escalation, persistent state-backend, declarative resource graphs, control-plane/GUI/scheduler, templating-engine — to do-one-thing-well tools or deliberately not-built. Net: Dorc stays "not an orchestrator," keeps the Ansible-loved sacred cows (agentless-ssh, idempotence, low-barrier, heterogeneity), targets the top user-study pains (control-flow expressivity, debugging/observability, deletion/drift via derived state, YAML/Jinja deletion, parallelism-from-dependencies), and avoids Terraform's central-state liability by construction.
